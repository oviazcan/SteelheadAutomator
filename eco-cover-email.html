A few points up front, because I think they speak directly to the concerns you raised:

  1. How the tools touch Steelhead. Everything runs inside the signed-in user’s own browser session, with that user’s existing permissions — the tools do only what the same person could do by hand in the UI. No admin or elevated access, and no private/undocumented channel: they reuse the same platform operations the Steelhead web app itself issues, so your server-side validation and safeguards stay fully in effect. We deliberately keep request volume conservative (small batches, capped concurrency) to be gentle on the platform.
  2. Read vs. write. A good share are read-only or merely pre-fill fields the user then reviews and saves. The large majority of the writes are reversible (we archive, not hard-delete) and carry an audit record (who / when / what). Only three tools perform higher-impact operations (an inventory reset, a process-tree rewrite, and an empty-folder cleanup) — each gated behind an explicit confirmation step. The matrix flags every tool’s access level, so the real footprint is visible per tool, including those three.
  3. On breakage when Steelhead updates. You’re right that these can break when the platform changes — that’s inherent to operating outside the native API. We mitigate it with an automated daily check that detects those changes proactively, usually before a user hits an error, and the fixes are typically quick.
  4. On the emergency-migration scenario. I understand the concern in principle, but I’d gently note that these tools use sanctioned platform operations, write reversibly in the large majority of cases, leave an audit trail, and gate the few higher-impact operations behind explicit confirmation — so that catastrophic-repair scenario doesn’t fit how they actually behave.
