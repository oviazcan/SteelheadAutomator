01El motor y la fotografía

Cómo existe una copia del ERP fuera del ERP, y por qué correr un reporte ahí es seguro por diseño.

Snapshot / fotografía semanal

Una copia congelada de la base de datos de Steelhead, guardada en un solo archivo .duckdb. Se genera automáticamente cada lunes a las 07:00 (el schedule scripts/weekly_snapshot.sh), y es sobre esa copia —no contra el ERP en vivo— que corren todos los reportes en local.

Metáfora: la fotografía semanal de la planta. No ves la planta en tiempo real; ves el retrato más reciente que se tomó de ella.

DuckDB

El motor de base de datos que lee el archivo .duckdb y ejecuta el SQL de cada reporte. La copia local de este repo corre DuckDB v1.2.2; el servidor de Steelhead que ejecuta los reportes publicados corre DuckDB v1.5.2 — dos versiones del mismo motor, con diferencias de comportamiento que sí importan (ver HUGEINT más abajo).

Solo lectura (read-only)

La copia local se abre siempre en modo solo lectura: ningún reporte puede hacer CREATE, INSERT, UPDATE ni ninguna otra escritura contra ella. Es estructuralmente imposible dañar los datos del ERP corriendo un reporte, porque el reporte nunca tiene permiso para escribir.

Metáfora: la foto con candado: se mira, no se altera.

pull / push

pull baja del ERP lo que el repo necesita: reportes existentes, el snapshot semanal, las vistas Perspective. push sube de vuelta el SQL o la vista ya escritos o corregidos, para que aparezcan publicados dentro de Steelhead.

Metáfora: pull = recoger las fotos reveladas del laboratorio; push = colgar el reporte en el tablero de la línea.

Banco de pruebas vs. línea real

Correr un reporte en local (contra el snapshot, con run_query.py) es el banco de pruebas: rápido y sin riesgo. Correrlo dentro de Steelhead —server-side, sobre DuckDB 1.5.2, con la sustitución real de variables del usuario— es la línea real, y ahí es donde aparecen ciertas trampas (HUGEINT, el dropdown 'null', el huso horario) que el banco de pruebas no reproduce. Por eso todo cambio de fondo se verifica en vivo antes de darlo por bueno.

Metáfora: el banco de pruebas no reproduce todas las condiciones de la línea real; algunas fallas solo aparecen al calibrar en producción.

02Los objetos de un reporte

La jerarquía con la que se organiza todo lo que ve un Key User dentro de Steelhead: del dato crudo al tablero directivo.

Reporte

El SQL fuente: trae todas las columnas relevantes de un dominio de negocio (una orden de trabajo, una factura, un movimiento de inventario…). Un reporte no decide cómo se ve esa información en pantalla — eso lo define cada una de sus vistas.

Metáfora: el cubo de datos.

Vista (Perspective)

Una forma de mirar el cubo de un reporte: una tabla (Datagrid) o una gráfica (barras, líneas, mapa de calor, treemap…) con sus propias columnas, agrupamientos, orden y filtros. Un mismo reporte puede tener varias vistas; se nombran con el sufijo decimal de su reporte padre — RPT-DEP-TIPO-NN.MM.

Metáfora: la forma de mirarlo.

Dashboard

Un tablero que combina varias vistas —de uno o de distintos reportes— en ventanas cuyo tamaño y posición se ajustan libremente, para una lectura de un vistazo.

Metáfora: el tablero directivo.

Variable / dropdown ($Var)

Un filtro configurado sobre un reporte (turno, línea, supervisor, cliente…) que aparece en la UI de Steelhead como un menú desplegable en "Variables ▾". El valor que el usuario elige se inyecta en el SQL en el lugar del placeholder — $Turno, $Linea, $Supervisor. Cuando el dropdown se deja sin seleccionar nada, ver la trampa Dropdown 'null' en el grupo 04.

Perspective

El motor de visualización de Steelhead sobre el que corren todas las vistas — tablas y alrededor de una docena de tipos de gráfica (mapa de calor, barras, líneas, dispersión, treemap, sunburst, velas japonesas, entre otros).

Placa RPT-DEP-TIPO-NN

La nomenclatura oficial de todo reporte: RPT (reporte oficial autorizado), DEP (departamento dueño, 3 letras — PRD, VTA, CAL…), TIPO (CAT catálogo de referencia, ANL analítica granular, o CTL control/cumplimiento) y NN (consecutivo de 2 dígitos dentro de ese departamento y tipo). El catálogo completo vive en el Inventario de Reportes.

Metáfora: la placa/matrícula del reporte — identificas cualquiera sin abrirlo, igual que identificas una tina por su placa sin meter la mano.

Version Control

El historial de versiones del SQL de un reporte que Steelhead guarda automáticamente cada vez que se publica un cambio (push). Permite comparar, dentro de la propia UI de Steelhead, el SQL viejo contra el nuevo lado a lado.

Carpeta y permisos (folder / managedPermissions)

Dónde vive un reporte dentro del menú de Steelhead (su carpeta) y quién puede verlo (sus permisos gestionados, p. ej. EXPORT_CSV). Todo reporte oficial debe terminar en su carpeta correcta con permisos validados — ninguno queda "sin carpeta" tras publicarse.

03La conexión con el ERP

Cómo el repo le habla a Steelhead sin ser parte de Steelhead: dos plantas, una llave, y las credenciales que la sostienen.

TLC / MTY (dominio / tenant)

Las dos plantas de Ecoplating, cada una su propio dominio (tenant) dentro de Steelhead: TLC es Tlalnepantla, la planta veterana con años de historial de datos; MTY es Monterrey, la planta de arranque más reciente. Todo cambio de reporte se aplica siempre a ambos dominios en la misma sesión — nunca a uno solo.

Metáfora: plantas gemelas: la veterana y la de arranque.

Hash / persisted query (APQ)

Steelhead corre su GraphQL en modo APQ-only (Automatic Persisted Queries): el servidor no acepta el texto de la consulta, solo un hash sha256 que la identifica de antemano. Sin el hash correcto ya registrado, el servidor simplemente no sabe qué operación ejecutar.

Metáfora: la llave que abre una operación del ERP.

Rotación de hash

Cada vez que Steelhead despliega una actualización, puede invalidar los hashes vigentes de ciertas operaciones. La señal es el error HTTP 400 "Must provide a query string": la llave vieja ya no abre esa cerradura, y hay que capturar la nueva desde el tráfico real del navegador antes de seguir operando ese endpoint.

Metáfora: Steelhead cambia la cerradura.

Cookie / refresh token

Las credenciales con las que los scripts del repo se autentican contra Steelhead: una cookie de sesión para ciertos endpoints REST, y un refresh token que renueva el JWT usado en /graphql. Se mantienen vigentes de forma automática mientras el schedule semanal corra al menos una vez cada 30 días; si pasa más tiempo sin correr, hay que repegar la cookie a mano.

04Las trampas técnicas

Bugs ya mordidos, ya diagnosticados y ya corregidos — documentados aquí para que no vuelvan a sorprender a nadie.

Ventana de fechas ($start / $end · "Use date range")

El filtro de rango que Steelhead inyecta como $start/$end cuando un reporte tiene activado "Use date range". Llega como timestamp UTC, no como fecha plana — truncarlo sin normalizar a hora local puede correr la cota superior un día de más.

Detalle completo del bug (tocó 53 reportes) y el patrón de corrección en la Ficha profunda: La ventana de fechas.

HUGEINT

Un entero de 128 bits que produce DuckDB al sumar columnas de conteo (típicamente SUM(part_count)). Es válido en local, pero el servidor de Steelhead (DuckDB 1.5.2) no puede serializarlo a JSON: el reporte corre perfecto en el banco de pruebas y muestra "This report did not return any data" en la línea real. Se corrige forzando CAST(SUM(...) AS BIGINT) en el SELECT final.

Dropdown 'null'

Cuando un usuario deja un filtro de dropdown sin seleccionar nada, Steelhead no manda el placeholder $Var ni una cadena vacía: inyecta literalmente el texto 'null' (en minúsculas). Si el SQL del reporte no contempla ese caso, el WHERE termina filtrando por la palabra "null" y el reporte devuelve cero filas en vivo — aunque en local, sin ese reemplazo, muestre miles.

Cliente combinado

El patrón obligatorio para la columna Cliente en todo reporte: nombre a secas si el cliente tiene una sola dirección de envío activa, o Cliente — Planta/Ciudad si tiene varias, para no confundir, por ejemplo, las distintas plantas de un mismo cliente que reciben material por separado.

Tipo de cambio (TC)

La tasa de MXN por USD que valoriza los reportes financieros. Se resuelve con una cadena de respaldo de 3 niveles: el histórico más cercano a la fecha objetivo, luego el tipo de cambio vigente del dominio (invertido), y por último el histórico más antiguo disponible — para que un reporte nunca se quede sin tipo de cambio.

Metáfora: la cadena de repuestos de respaldo del almacén: si no está el repuesto exacto, se usa el siguiente disponible en la jerarquía.

CTE

Un bloque WITH ... AS (...) de SQL que nombra un paso intermedio de una consulta. Es la convención obligatoria del repo para toda lógica de varios pasos: nada de tablas temporales, nada de subconsultas anidadas difíciles de leer.

microdólares / microcantidad

La unidad interna en la que Steelhead guarda montos y cantidades. Hay que dividir entre 1,000,000 para obtener el dólar o la pieza real: montos en microdollars, cantidades en microquantity.
