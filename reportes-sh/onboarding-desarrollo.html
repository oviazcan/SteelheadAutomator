Onboarding de Desarrollo

Monta el entorno y echa a andar el pipeline de Reportes SH en una máquina nueva — Windows o Mac — para poder mantenerlo y evolucionarlo.

Desarrollo Documento perenne

00 · En resumen

Vas a clonar el repo, instalar tres tipos de dependencias de Python, configurar un .env con credenciales del ERP Steelhead, bajar tu propia "fotografía" DuckDB, y correr un query de prueba. Con eso, tu máquina queda lista para consultar datos, correr las pruebas y — cuando toque — publicar reportes en los dos tenants (TLC y MTY). Cuenta ~30–45 minutos para tener verify_handover.py en verde (clonar, instalar, configurar .env, autenticarte), más ~10–20 minutos extra si generas tu propia fotografía DuckDB desde cero (~5–10 min por tenant). Fuente: ONBOARDING.md §2 y TRANSFER.md §2.1 de la raíz del repo.

01Panorama del repo

Cuatro líneas antes de tocar nada — el modelo mental completo vive en la Presentación 101, no aquí.

Reportes SH es el pipeline que conecta el ERP Steelhead (multi-tenant: TLC y MTY) con un layer de SQL refactorizado y vistas Perspective oficiales. Cada semana baja una fotografía completa de los datos del ERP a un archivo .duckdb local; sobre esa fotografía corren los query_*.sql del repo (~89 reportes oficiales, código RPT-<DEP>-<TIPO>-<NN>); y los scripts de scripts/ se encargan de bajar (pull) y subir (push) reportes y vistas de vuelta a Steelhead, manteniendo TLC y MTY sincronizados. El repo vive en la Capa 5 de la arquitectura Ecoplating — ver docs/gobernanza.md.

Trampa de entorno conocida en Mac (intérprete equivocado)

Si python3 en tu Mac apunta a una instalación muy reciente (por ejemplo Homebrew con una versión nueva de Python) y ves ModuleNotFoundError: No module named 'duckdb' al correr cualquier script del repo, es porque el paquete duckdb no está instalado ahí — o no hay wheel publicado todavía para esa versión de Python. La salida documentada en CLAUDE.md: usar el intérprete de sistema en /usr/bin/python3 (el que trae macOS con las Command Line Tools, típicamente 3.9.6), anteponiéndolo a cualquier comando del repo — por ejemplo /usr/bin/python3 run_query.py query.sql. Aplica igual a los scripts de scripts/.

03Clonar e instalar

  1. 1

    Clonar el repo

    En Windows, ábrelo directo en Git Bash y clona en una ruta corta y sin espacios si puedes (Git Bash acepta forward slashes: C:/repos/Reportes-SH). Con espacios funciona, pero hay que entrecomillar todos los comandos.

  2. 2

    Paquetes runtime de scripts/

    Los scripts de sincronización (pull_reports.py, push_query.py, regenerate_duckdb.py, run_query.py…) necesitan tres paquetes que no vienen en requirements.txt — se instalan aparte, tal como documenta ONBOARDING.md §2.2.

  3. 3

    Herramientas de documentación y pruebas (opcional según lo que vayas a tocar)

    requirements.txt / requirements-dev.txt cubren el renderizador de estos documentos (scripts/render_docs.py) y la suite de pytest — instálalos si vas a correr pruebas o a tocar docs/training/.

# macOS / Linux
git clone https://github.com/oviazcan/Reportes-SH.git "Reportes SH"
cd "Reportes SH"

# paso 2 — paquetes runtime de scripts/ (NO están en requirements.txt)
python3 -m pip install --user requests duckdb cryptography

# paso 3 — herramientas de documentación + pruebas (opcional)
python3 -m pip install -r requirements.txt
python3 -m pip install -r requirements-dev.txt
# Windows (Git Bash) — mismo flujo, sustituyendo python3 → python
git clone https://github.com/oviazcan/Reportes-SH.git "Reportes SH"
cd "Reportes SH"

python -m pip install --user requests duckdb cryptography
python -m pip install -r requirements.txt
python -m pip install -r requirements-dev.txt

ONBOARDING.md (última revisión 2026-06-16) dice: "El repo no usa requirements.txt porque el set es mínimo y estable". Eso ya no es exacto — requirements.txt y requirements-dev.txt sí existen hoy (se agregaron después, commit "build: deps para conversión MD→HTML"), pero cubren únicamente el renderizador de documentación y las pruebas — no los paquetes runtime de scripts/ (requests, duckdb, cryptography), que siguen instalándose aparte como en el paso 2 de arriba. Vale la pena avisarle a quien mantenga ONBOARDING.md para que se actualice esa línea.

04Configurar el .env

Todos los secretos del repo viven en .env — nunca en código, nunca en un commit.

cp .env.example .env

Abre .env y llena las variables. Ninguna se genera sola — todas salen de iniciar sesión en el ERP y capturarlas desde DevTools del navegador; el procedimiento exacto (con los dos modos posibles, cookie o refresh token) está en scripts/README.md § Setup.

VariableQué esNotas
STEELHEAD_COOKIE_STRINGHeader Cookie: de la sesión del navegador en app.gosteelhead.com.Secreto. Se auto-renueva cada vez que un script corre (keep-alive) y se reescribe sola de vuelta al .env. Usa una de las dos credenciales de auth (esta o la de abajo).
STEELHEAD_REFRESH_TOKENOAuth2 refresh token de Authentik (auth.gosteelhead.com).Secreto. Estrategia renewable: cada uso lo renueva, no caduca mientras se use.
STEELHEAD_DOMAIN_TLC / STEELHEAD_DOMAIN_MTYnanoID de cada tenant (lo que Steelhead manda como ?domainNanoId=…).No es secreto — es un identificador de tenant, ya viene precargado en .env.example.
STEELHEAD_DEFAULT_DOMAINCuál dominio usar si un script no recibe --domain.Texto: tlc o mty.

Nunca expongas valores reales

.env está en .gitignore — jamás lo commitees, ni pegues su contenido en un chat, ticket o log. Si un secreto se filtra (cookie o refresh token), rótalo de inmediato: cierra sesión en Steelhead/Authentik, vuelve a entrar, y recaptura el valor nuevo (procedimiento en ONBOARDING.md §3).

05Primer humo: correr un query local

La prueba mínima de que tu entorno sirve: correr cualquier query_*.sql del repo contra el .duckdb más reciente.

# macOS / Linux
python3 run_query.py query_facturacion.sql

# si truena con ModuleNotFoundError: No module named 'duckdb' (trampa del §02)
/usr/bin/python3 run_query.py query_facturacion.sql

# Windows (Git Bash)
python run_query.py query_facturacion.sql

run_query.py auto-detecta el .duckdb más reciente del dominio default en steelhead_snapshot/<dominio>/duckdb/, y abre la conexión en modo read-only — ningún query del repo puede escribir sobre la fotografía. Debe imprimir filas de resultado sin errores.

Si es tu primera vez en la máquina

Un clon nuevo no trae ningún .duckdb (están en .gitignore por su tamaño). Si run_query.py no encuentra ninguno, es normal — genera tu primera fotografía con el §06 de abajo y vuelve a correr el query.

06Bajar la fotografía (pull / regen)

Dos comandos generan tu copia local: uno baja los reportes y vistas tal como viven hoy en Steelhead, el otro genera el .duckdb con los datos.

  1. 1

    Bajar reportes y vistas (snapshot reverso)

    python3 scripts/pull_reports.py --domain tlc (o --domain all para los dos tenants). Deja steelhead_snapshot/<dom>/ con index.json, reports/ y perspective_configs/. Es idempotente y rápido (~30s por tenant). Si termina en Snapshot en steelhead_snapshot/tlc/, la conexión está OK.

  2. 2

    Generar el .duckdb con los datos

    python3 scripts/regenerate_duckdb.py --domain all --download --decrypt. Toma ~5–10 minutos por tenant: encola la generación en Steelhead, espera a que termine, descarga el archivo cifrado y lo descifra. Deja el resultado en steelhead_snapshot/<dom>/duckdb/*.duckdb — con esto ya corre el §05 de arriba.

Automatizar el snapshot semanal (opcional, recomendado)

scripts/weekly_snapshot.sh encadena regenerate_duckdb.py --domain all --download --decrypt --prune (conserva las 2 generaciones más recientes por dominio) + schema_diff.py --domain all --json, y deja el log rotado en logs/weekly_snapshot.log. Autodetecta el repo root y el intérprete Python (python3 → python → py) — no hay paths que editar a mano.

macOS — launchd

cp scripts/launchd/com.ecoplating.steelhead-weekly-snapshot.plist ~/Library/LaunchAgents/
launchctl load ~/Library/LaunchAgents/com.ecoplating.steelhead-weekly-snapshot.plist

El plist trae el path absoluto del repo tal como estaba en la máquina donde se armó — edítalo si tu copia vive en otra ruta.

Windows — Programador de tareas (Task Scheduler)

No hay un plist equivalente en Windows; se configura una tarea nueva a mano en taskschd.msc: una tarea avanzada (no "Basic Task"), disparador semanal, y una acción "Start a program" que invoque bash.exe de Git for Windows con el argumento -lc "cd '<ruta del repo>' && bash scripts/weekly_snapshot.sh". El paso a paso completo con cada pantalla del asistente está en ONBOARDING.md §2.7.2 — remítete ahí para no perder ningún clic.

Por qué corre bien en ambos SO sin tocar el script

weekly_snapshot.sh detecta solo qué intérprete de Python hay disponible — no hace falta una versión distinta del script por sistema operativo, solo que el programador de tareas invoque un shell tipo bash (Git Bash en Windows).

07Publicar un reporte (resumen)

El detalle operativo día a día — con sus reglas de oro — vive en el Manual del Administrador (en preparación al momento de escribir este documento) y en docs/runbook.md. Aquí solo el resumen del flujo, para que sepas qué esperar la primera vez.

  1. 1

    Re-pull del dominio destino

    python3 scripts/pull_reports.py --domain all — refresca folderId y permisos vigentes antes de tocar nada. Saltarse este paso es la causa más común de que un push "regrese" un reporte a la carpeta raíz o borre sus permisos.

  2. 2

    Editar el SQL o la vista

    El query_*.sql del repo, o el .config.json de la vista Perspective en perspective_views/<código>/.

  3. 3

    Dry-run

    python3 scripts/push_query.py --domain all --name "<nombre exacto>" query_x.sql (sin --commit) — revisa que la mutación se vea bien antes de aplicarla.

  4. 4

    Push real

    Mismo comando, agregando --commit. Si el SQL usa $start/$end, el push no activa el interruptor "Use date range" — hay que encenderlo a mano en la UI de Steelhead.

  5. 5

    Verificar EN VIVO

    El re-pull inmediato todavía puede mostrar la versión vieja (latencia del backend). La verdad está en lo que Steelhead tiene cargado ahora mismo, no en el archivo recién descargado — ver la ficha de la ventana de fechas para el porqué exacto de esta regla.

Un error que vas a ver tarde o temprano

Si un push falla con algo como PERSISTED_QUERIES_NOT_FOUND ("Must provide a query string"), es que Steelhead cambió la cerradura de esa operación GraphQL — el hash guardado en scripts/steelhead_client.py ya no abre. Hay que recapturar el hash nuevo desde DevTools (la "llave" nueva) y actualizar el diccionario PERSISTED_QUERIES. El detalle vive en docs/sincronizacion_steelhead.md y en la tabla de troubleshooting de ONBOARDING.md §8.

08Pruebas y verificación de handover

Suite de pruebas (pytest)

# una sola vez
python3 -m pip install -r requirements-dev.txt

# desde la raíz del repo (conftest.py agrega el repo root al sys.path)
python3 -m pytest tests/ -v

Son pruebas unitarias (unittest, corridas vía pytest) sobre lógica pura — parseo de códigos RPT-…, aliases, diffs de vistas, manifests, plugins Perspective, rotación de datos, y el renderizador de estos mismos documentos (test_render_docs.py). No tocan Steelhead ni requieren .env: corren igual sin credenciales ni conexión.

Verificación de handover (verify_handover.py)

python3 scripts/verify_handover.py

Corre 7 checks de salud, en orden, e imprime OK/FAIL/SKIP por cada uno:

  1. .env y variables — existe y trae las variables mínimas (dominios + al menos una credencial de auth).
  2. Paquetes Python — requests, duckdb, cryptography importables.
  3. Credenciales Steelhead — keep-alive contra TLC y MTY.
  4. Pull dry-run — la consulta AllReports baja sin errores.
  5. DuckDB local — hay snapshot y se puede consultar en modo read-only.
  6. Auditoría DEP-nombre vs DEP-header — debe dar 0 mismatches en cada dominio.
  7. Estructura del repo — scripts y docs esperados presentes.

El exit code es el número de FAILs. Si imprime ✓ Setup validado. Listo para operar., el entorno quedó funcional de punta a punta.
