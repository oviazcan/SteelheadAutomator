Cómo se opera Reportes SH en productivo

La fotografía semanal, publicar un reporte, sincronizar las plantas gemelas, credenciales y qué hacer cuando Steelhead cambia la cerradura. Nivel operación — el montaje del entorno de desarrollo vive en ONBOARDING.md.

Jefe de TI Documento perenne

En pocas palabras

Reportes SH corre en un ciclo semanal: recoger la fotografía del ERP (snapshot DuckDB), correr y editar SQL en el banco de pruebas local, y publicar de vuelta a Steelhead en las dos plantas gemelas (TLC y MTY). Este manual documenta las cinco piezas que hay que mantener vivas: el snapshot automático, el flujo de publicación, la sincronización entre plantas, las credenciales, y qué hacer cuando Steelhead rota los hashes de sus operaciones GraphQL.

01El pipeline de un vistazo

Todo lo que haces como administrador es una vuelta de esta misma rueda. Entender el ciclo completo hace que cada comando individual tenga sentido.

Las secciones siguientes desarman cada movimiento de esta rueda: cómo correr SQL en el banco de pruebas (§02), cómo se toma la fotografía (§03), cómo se publica (§04), cómo se mantienen las dos plantas gemelas al día (§05), y qué hace que la rueda se atore — credenciales (§06) y hashes rotados (§07).

02El entorno y correr SQL local

Antes de tocar cualquier reporte, todo pasa por el banco de pruebas: una base DuckDB local, en modo solo-lectura, que es la fotografía más reciente del ERP.

# dominio default (el que diga STEELHEAD_DEFAULT_DOMAIN en .env, o tlc)
python3 run_query.py query_facturacion.sql

# dominio explícito
python3 run_query.py query_facturacion.sql --domain mty

# override de la ruta del .duckdb (para pruebas puntuales)
python3 run_query.py query_facturacion.sql --db /ruta/explícita.duckdb

run_query.py auto-detecta el .duckdb más reciente en steelhead_snapshot/<dominio>/duckdb/ — no hay que apuntarlo a mano tras cada fotografía semanal. Abre la base en modo read-only: no hay CREATE/INSERT posible desde aquí, es intencional.

La trampa del intérprete

El paquete Python duckdb vive instalado en /usr/bin/python3 (Python 3.9.6), NO en el python3 que resuelve el PATH por Homebrew (3.14). Si corres python3 run_query.py ... y truena con ModuleNotFoundError: No module named 'duckdb', no es que falte instalar nada — es el intérprete equivocado. Usa /usr/bin/python3 run_query.py .... Aplica igual a los scripts de scripts/ (pull_reports.py, push_query.py, etc.) si presentan el mismo síntoma.

Una versión de DuckDB distinta en cada lado

El banco de pruebas local corre DuckDB 1.2.2; el servidor de Steelhead que ejecuta el reporte publicado corre 1.5.2. La mayoría del tiempo es transparente, pero hay diferencias reales de comportamiento entre versiones (por ejemplo, la serialización de enteros grandes a JSON). Es la misma lógica de banco de pruebas contra línea real que aparece en la §08 de incidentes — un reporte que corre perfecto en local puede fallar publicado por esta razón.

03La fotografía semanal

El snapshot que alimenta todo el banco de pruebas se toma solo, cada semana. Rara vez hay que intervenir, pero hay que saber verificarlo y saber correrlo a mano.

Cómo corre (automático)

Un launchd agendado dispara scripts/weekly_snapshot.sh cada lunes 07:00 (si la Mac está dormida a esa hora, corre al despertar). El wrapper hace dos cosas, en orden, por los dos dominios:

  1. regenerate_duckdb.py --domain all --download --decrypt --prune — encola la generación en Steelhead, espera a que termine (polling), descarga el archivo cifrado desde S3, lo descifra (AES-256-CBC + gunzip) y conserva solo las 2 generaciones más recientes por dominio (--prune).
  2. schema_diff.py --domain all --json — compara el esquema (tablas/columnas) entre los dos snapshots más recientes y vuelca un log JSON versionado en logs/schema_diff/<ts>_<dom>.diff.json.

La salida completa queda en logs/weekly_snapshot.log, y el wrapper manda una notificación nativa de macOS (sonido Glass si terminó bien, Basso si falló) para que no dependas de ir a revisar el log para enterarte.

Cómo verificar que corrió

  • Abre logs/weekly_snapshot.log y busca el bloque con la fecha más reciente — cada corrida agrega su propio bloque, no lo sobrescribe.
  • El bloque termina con Exit code: 0 si ambos dominios (TLC y MTY) generaron su .duckdb sin problema. Cualquier otro código indica falla — ver §08.
  • La notificación de macOS es la señal más rápida en el momento; el log es la fuente de verdad si la notificación se perdió (Mac ocupada, pantalla apagada).

Cómo correrla a mano

bash scripts/weekly_snapshot.sh

Corre el mismo flujo completo (regen + diff + log + notificación) fuera del calendario. Vale la pena hacerlo cuando:

  • Ha pasado más de 20 días sin actividad en el repo — la cookie de sesión dura 30 y conviene refrescarla antes de que caduque.
  • Steelhead acaba de desplegar un cambio que pudo tocar la generación de DuckDB.
  • Necesitas un .duckdb fresco ya mismo para un análisis puntual, sin esperar al lunes.

Comandos de operación del schedule

# Estado / último run / próximo disparo
launchctl print gui/501/com.ecoplating.steelhead-weekly-snapshot \
  | grep -E "state|runs|last exit"

# Disparar ahora, saltándose el calendario
launchctl kickstart -k gui/501/com.ecoplating.steelhead-weekly-snapshot

# Pausar / reanudar
launchctl disable gui/501/com.ecoplating.steelhead-weekly-snapshot
launchctl enable  gui/501/com.ecoplating.steelhead-weekly-snapshot

# Re-instalar tras editar el plist
launchctl bootout  gui/501/com.ecoplating.steelhead-weekly-snapshot
launchctl bootstrap gui/501 \
  /Users/oviazcan/Library/LaunchAgents/com.ecoplating.steelhead-weekly-snapshot.plist

El plist vive en ~/Library/LaunchAgents/com.ecoplating.steelhead-weekly-snapshot.plist; hay una copia versionada en scripts/launchd/ para poder reproducirlo en otra máquina.

Revisión mensual

Una vez al mes vale la pena: (1) confirmar que cada corrida del último mes cerró en Exit code: 0 en los dos dominios; (2) auditar logs/schema_diff/*.diff.json por cambios en tablas núcleo (invoice, work_order, part_number…) que puedan afectar algún query_*.sql; (3) confirmar que la cookie sigue vigente (ls -l .env, mtime ≤30 días).

04Publicar un reporte, paso a paso

"Colgar en el tablero" un reporte nuevo o modificado tiene un orden fijo. Saltarse el primer paso es la causa más común de folders y permisos borrados.

  1. 1

    Re-pull del dominio destino

    python3 scripts/pull_reports.py --domain <d> (o --domain all). Trae el folderId y los permisos vigentes al snapshot local. Sin este paso, el push puede sobrescribirlos con datos viejos — es obligatorio, no opcional.

  2. 2

    Editar el SQL

    Sobre el query_*.sql del repo, siguiendo las convenciones del proyecto (CTEs, json_extract_string, microdollars ÷ 1e6, patrón de $start/$end).

  3. 3

    Subir el SQL

    push_query.py --domain <d> --name "<título>" query_x.sql actualiza un reporte que ya existe (dry-run por default; agrega --commit para ejecutar). Si el reporte es nuevo, usa create_report.py --name "..." --domain <d> --folder-id <id> --permissions <lista> --commit (mutación con reportId: null).

  4. 4

    Vistas Perspective

    push_query.py invoca automáticamente push_view.py por cada vista hija activa del reporte tras subir el SQL. Si necesitas subir el SQL sin tocar las vistas, agrega --no-views.

  5. 5

    Variables / dropdowns

    push_query.py no sincroniza variables. Los dropdowns ($Turno, $Linea, $Supervisor…) se crean, editan o borran aparte con push_variable.py (--create / update / --delete).

  6. 6

    "Use date range" si el SQL usa $start/$end

    Este flag tampoco viaja con el push de SQL. Actívalo a mano en la interfaz de Steelhead la primera vez, o vía set_date_range.py --domain <d> --name "..." --on --commit (mutación UpdateReportDateRange). Una vez encendido, queda persistido en pushes futuros.

  7. 7

    Verificar EN VIVO

    Consulta AllReports contra el ERP (nodes[].latestQuery, useDateRange, reportVariablesByReportId) para confirmar que el cambio quedó. Nunca contra un re-pull inmediato: el backend de Steelhead tarda en reflejar el cambio, y un re-pull recién hecho puede mostrar todavía la versión anterior.

Si el folderId sigue en null tras el re-pull

Avisa antes de publicar. Probablemente el snapshot está obsoleto, o el reporte de verdad vive en la raíz (root) de Steelhead. Empujar sin resolver esto puede mover un reporte oficial fuera de su carpeta.

05Sincronizar las plantas gemelas

TLC y MTY son plantas gemelas: mismo catálogo de reportes, mismo SQL — pero credenciales, IDs internos y folders distintos. Mantenerlas sincronizadas es una regla dura del proyecto, no una sugerencia.

Todo cambio de reporte o SQL en un dominio debe aplicarse también al otro, en la misma sesión de trabajo — es una autorización permanente, no hay que pedir confirmación adicional cada vez. Lo que sí exige atención:

  • Portabilidad por nombre. El SQL nunca debe depender de IDs internos de un dominio (dimension_id = 349, unit_id = 3969…) — esos IDs difieren entre TLC y MTY. Resolver siempre por nombre (acct_dimension.name = 'Línea', unit.name LIKE 'KGM%'). Auditar esto antes de cada push.
  • Reportes nuevos en ambos dominios. Se crean con reportId: null en cada uno por separado; el id_in_domain y el reportId serán distintos, pero nombre, folder y permisos deben coincidir.
  • Verificar cobertura tras sincronizar. Los IDs y hasta los nombres pueden existir en un dominio y no en el otro — repórtalo si encuentras diferencias.

Sembrar MTY desde cero (tercer tenant o catálogo faltante)

Cuando un dominio arranca sin reportes, la siembra es una secuencia de tres scripts, en este orden:

# 1. SQL padre — crea cada query_*.sql como reporte nuevo si su código RPT no existe ahí
python3 scripts/push_to_mty_batch.py --target mty --commit
python3 scripts/pull_reports.py --domain mty

# 2. Vistas Perspective — resuelve el id del padre recién creado y las siembra
python3 scripts/seed_views_mty.py --target mty --commit

# 3. Folder + permisos — asigna carpeta por TIPO y copia permisos del equivalente TLC
python3 scripts/sync_mty_meta.py --target mty --commit

El paso 3 resuelve la carpeta destino por el TIPO del código RPT: ANL → 456, CAT → 455, CTL → 457 (IDs de folder en MTY). Conviene re-pull entre pasos, sobre todo entre el 1 y el 2, porque el batch genera IDs nuevos que el manifest de vistas necesita.

Crítico — no metas una carpeta a ~50 reportes

Steelhead pagina reportsByFolderId con onFull: ERROR. Cuando una sola carpeta llega a ~50 reportes, el query completo aborta — y ese mismo query es el que arma el listado de reportes personalizados en la interfaz (View Reports → Custom Reports). El resultado no es un error silencioso: todos los reportes personalizados desaparecen de ese listado para todos los usuarios del dominio (los dashboards que referencian el reporte por ID siguen funcionando; lo que se rompe es el listado de navegación). También revienta pull_reports.py.

Incidente real (2026-06-09): publicar un reporte llevó la carpeta ANL. Fuente Analítica de TLC de 49 a 50 reportes → listado vacío en TLC. El arreglo fue mover ese reporte a root (folderId: null) para que la carpeta volviera a 49. Regla operativa: antes de crear o mover un reporte a una carpeta, confirma que se quede por debajo de ~49. Si una carpeta ANL está al borde, hazle espacio (archivar deprecados) o crea una segunda carpeta antes de seguir metiendo reportes.

06Credenciales y accesos

Steelhead usa OAuth2 PKCE contra Authentik (auth.gosteelhead.com) más una sesión de cookie que se renueva sola mientras la uses con regularidad.

Las credenciales viven en .env (gitignorado — nunca se comitea):

  • STEELHEAD_COOKIE_STRING — la sesión pegada desde DevTools. El cliente la rota automáticamente en cada keep-alive y reescribe el valor nuevo de vuelta al .env; mientras corras algo al menos cada 30 días, nunca caduca.
  • STEELHEAD_REFRESH_TOKEN — modo alterno vía OAuth, no requiere renovación manual.
  • STEELHEAD_DOMAIN_TLC / STEELHEAD_DOMAIN_MTY — el nanoID de cada tenant. STEELHEAD_DEFAULT_DOMAIN decide cuál usan los scripts sin --domain explícito.

Un header por endpoint, no al revés

El header x-steelhead-idp-token (el JWT) va solo en /graphql. El endpoint /api/files/<archivo> exige cookie únicamente — si le mandas el JWT junto con la cookie, responde 401 aunque la cookie sea válida. Mezclarlos es el bug más común al depurar 401/403: el JWT viene de un refresh_token reciente y la cookie de un login anterior del navegador; si Steelhead los ve incongruentes, rechaza la llamada.

Cuando algo da 401/403: corre python3 scripts/steelhead_auth.py para ver el estado de la cookie y del access_token refrescado. Si la cookie realmente expiró, hay que volver a pegarla desde DevTools — el procedimiento completo (paso a paso, con capturas de dónde mirar en el navegador) vive en ONBOARDING.md §3, porque es un procedimiento de puesta a punto de máquina, no de operación diaria.

07Cuando Steelhead cambia la cerradura: hashes rotados

Steelhead corre su /graphql en modo APQ-only (Automatic Persisted Queries): el cliente nunca manda la consulta completa, solo su hash — como enseñar una llave en vez de describir la cerradura. Cuando Steelhead se actualiza, a veces cambia la cerradura: rota el hash de una operación, y la llave que teníamos guardada deja de abrir.

La metáfora

Cada operación de GraphQL (AllReports, CreateUpdateReportWithPermissions…) tiene su propia llave: un hash sha256 que Steelhead reconoce sin necesitar la consulta escrita completa. El repo guarda el llavero completo en PERSISTED_QUERIES, dentro de scripts/steelhead_client.py. Cuando Steelhead despliega una versión nueva, puede cambiar la cerradura de algunas puertas (operaciones) y dejar otras igual — no es todo o nada.

El síntoma es inconfundible: /graphql responde HTTP 400 {"message":"Must provide a query string."} (hash no registrado) o levanta PersistedQueryNotFound desde el cliente Python. Cuando lo veas, la llave vieja ya no sirve — toca recapturar la nueva.

Cómo se recaptura un hash

Estando logueado en app.gosteelhead.com, se intercepta window.fetch desde la consola del navegador (o vía mcp__claude-in-chrome) y se dispara en la interfaz la acción que use esa operación:

// Pegar en la consola de app.gosteelhead.com, logueado, SIN recargar después:
window.__cap = {};
const _f = window.fetch;
window.fetch = async function(...a){
  try{ const [u,o]=a; const url=typeof u==='string'?u:u&&u.url;
    if(url&&url.includes('/graphql')&&o&&o.body){
      for(const it of [].concat(JSON.parse(o.body))){
        const h=it.extensions&&it.extensions.persistedQuery&&it.extensions.persistedQuery.sha256Hash;
        if(it.operationName&&h) window.__cap[it.operationName]={hash:h, variables:it.variables};
      }}}catch(e){}
  return _f.apply(this,a);
};
// Leer: JSON.stringify(window.__cap.NombreDeLaOperacion)

El par operationName → hash capturado se pega en PERSISTED_QUERIES. Este método (interceptor en vivo) es más confiable que depender de localStorage.__sa_scan_backup (un scan que deja el SteelheadAutomator): algunas mutaciones — como CreateReportConfig / UpdateReportConfig — solo se registran cuando de verdad ejecutas la acción que las dispara (guardar una vista), y un scan de navegación normal no las captura.

Caso real — rotación parcial (2026-07-20)

Un deploy de Steelhead rotó los hashes de la capa de reportería (AllReports, GetReportingSettings, GetRecomputableAt, DuckDbContextQuery…) pero dejó intactos los de la capa operativa (p. ej. AllPartNumbers). Mientras no se recapturaron, quedaron afectados pull_reports.py, push_query.py, los dashboards, y la regeneración de DuckDB — por eso el snapshot semanal de esa fecha no generó un .duckdb nuevo. Se resolvió recapturando 15 hashes desde el __sa_scan_backup y verificando en vivo (pull de los dos dominios, push de reportes de prueba, regeneración exitosa).

Dependencia cruzada, con honestidad

La validación diaria de que las llaves sigan vigentes (un cron que corre cada mañana y avisa si algo rotó) vive en el repositorio hermano SteelheadAutomator, aunque usa las credenciales de este repo. No es parte de Reportes SH — es una dependencia externa que conviene tener presente si un día deja de correr.

08Incidentes y recuperación

El patrón que más se repite en este sistema es el mismo: algo se ve perfecto en el banco de pruebas local y falla en la línea real, publicado dentro de Steelhead. El caso de estudio de referencia para esta familia completa de bugs es la ficha profunda: La ventana de fechas — ahí quedó la disciplina de "verifica en vivo, no en el banco" que aplica a todo lo demás.

SíntomaCausa probableQué hacer
El reporte incluye un día de más en el límite superior del rango de fechas. $end llega en hora UTC; truncarlo "a secas" lee el día siguiente. Aplicar el patrón canónico (TRY_CAST(...AS TIMESTAMPTZ) AT TIME ZONE 'America/Mexico_City'). Ver ficha profunda: La ventana de fechas.
El reporte da 0 filas en vivo pero corre y devuelve datos bien en local. Alguna columna SUM/conteo de enteros devuelve HUGEINT; el servidor (DuckDB 1.5.2) no puede serializarlo a JSON. CAST(SUM(...) AS BIGINT) en el SELECT final de toda columna de conteo/suma.
El reporte filtra todo (0 filas) cuando un dropdown se deja sin selección. Steelhead inyecta la cadena literal 'null' (no vacío) cuando no hay selección; el patrón '$Var' LIKE '$%' no la detecta. Ampliar el hook: OR lower(TRIM('$Var')) IN ('null','none') tratado como sin-selección.
El listado de reportes personalizados aparece vacío para todo el dominio (View Reports → Custom Reports). Una carpeta llegó a ~50 reportes; reportsByFolderId aborta el query completo. Mover el reporte más reciente de esa carpeta a root o a otra carpeta; re-pull para confirmar que el listado volvió. Ver §05.
Una vista Perspective queda muda: [UnknownError] Unknown column "X" in group_by. Se cambió el SQL del reporte padre y Perspective sigue cacheando el esquema viejo en el cliente. Refrescar la pestaña del navegador. No requiere cambio de código.
El schedule semanal terminó con Exit code: 1. Ver el bloque más reciente de logs/weekly_snapshot.log: cookie expirada (401/403), hash rotado (PersistedQueryNotFound), o el job de generación se quedó colgado (timeout). Rotar la cookie (§06), recapturar el hash (§07), o re-correr el schedule si fue un timeout puntual.
Un push borró el folderId o los permisos de un reporte. Se publicó sin re-pull previo, o usando --report-id (que no carga la metadata completa). Revertir desde la UI de Steelhead (mover de regreso, repegar permisos) y re-pull para que el snapshot quede al día. La próxima vez: re-pull → --name → dry-run → --commit.

09Rollback

Publicar no es un camino de un solo sentido: Steelhead guarda cada versión y se puede volver atrás sin tocar el repo.

Cada push_query.py exitoso (mutación CreateUpdateReportWithPermissions) crea automáticamente una versión nueva visible en Version Control, dentro de la propia interfaz de Steelhead, sobre el reporte. El creatorId se llena solo con el usuario de la sesión — no hace falta mandarlo. Para volver a una versión anterior, se hace directamente desde ahí, sin script.

Una trampa de lectura, no de datos

El visor de diff lado a lado de Steelhead no siempre hace scroll automático de la columna izquierda hasta el final del SQL eliminado. Con una versión mucho más corta que la otra, puede parecer que la versión vieja "solo tenía unas líneas" cuando en realidad el cuerpo completo está más abajo. No es un bug de datos — solo hay que bajar manualmente en esa columna antes de concluir algo.

10Qué significa para ti

Jefe de TI · Administrador

Tu semana gira sobre este ciclo: confirma que el snapshot del lunes cerró en Exit code: 0, re-pull antes de cada push, verifica siempre EN VIVO (nunca contra el re-pull recién bajado), y recuerda que TLC y MTY se sincronizan en la misma sesión. Cuando algo falla de la nada, sospecha primero de la cookie o de un hash rotado — son las dos causas más comunes.

Key User · Consumidor

No necesitas correr nada de este manual. Si un reporte se ve raro (vacío, con un día de más, un dropdown que no filtra), repórtalo con el síntoma exacto — la tabla de la §08 casi seguro ya tiene el diagnóstico. El Manual del Operador (paquete de capacitación) es tu punto de entrada normal.
