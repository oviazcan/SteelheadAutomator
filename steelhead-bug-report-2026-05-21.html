We have hit three independent issues in the native quote editor that block manual editing of quote part-number prices. Bugs #1 and #2 are tightly coupled (the second is a downstream effect of the first); bug #3 is independent but lives in the same area.

Bug #1 — CRITICAL PartNumberDialogProcessQuery fails: allSpecFieldParamDropdowns pagination limit

Symptom

When opening a part-number line inside a quote for editing, the dialog that populates the price builder fails to load fully. The GraphQL operation PartNumberDialogProcessQuery (hash 01d7d09423a65ca3772be469cb87aa7b7703064a7e4931f1b90a7622cd0f94b2) returns HTTP 200 but with a GraphQL error in the response body:

{
  "errors": [{
    "message": "This field has hit the pagination limit and `onFull: ERROR` is set. Please use pagination when fetching from this field and then remove the `onFull` argument.",
    "locations": [{ "line": 146, "column": 3 }],
    "path": ["allSpecFieldParamDropdowns"]
  }]
}

The error fires consistently. Out of two consecutive opens we captured, both produced this error (errorCount = 2 in the captured session).

Root cause (based on the error message)

The PartNumberDialogProcessQuery selection set includes a child field allSpecFieldParamDropdowns that is being fetched without pagination, and the field is configured with onFull: ERROR. As our domain grew (we recently created several hundred new SpecFieldParam records via bulk operations), the total count of specFieldParamDropdowns in the domain exceeded whatever internal threshold the field enforces. The backend now refuses the fetch instead of returning the list truncated.

User impact: The dialog opens visually but key inputs (treatment selection dropdowns, default values, currency selectors) come up empty or stale. The user cannot edit a quote line in any meaningful way. This blocks all manual quote edits for any user in this domain until the query is patched, regardless of how the quote was created.
Suggested fix: Update the PartNumberDialogProcessQuery selection set to paginate allSpecFieldParamDropdowns (Relay-style first / offset or cursor), or fetch only the dropdowns referenced by the current part-number's specFields rather than the entire domain catalog. The onFull: ERROR argument should also be removed from this selection — surfacing a hard failure in a query that powers a critical UI dialog produces an unrecoverable state for the end user.

Bug #2 — HIGH SaveManyPartNumberPrices fails with generic error after Bug #1

Symptom

When the user attempts to save the quote line anyway (because Steelhead's UI does not surface the silent failure from Bug #1), the save mutation SaveManyPartNumberPrices (hash b3c7414179e41a07db3912deac48f0cd4bc7ac1569b9d6d1e3a505065ecdfdbc) returns HTTP 200 with:

{
  "errors": [{
    "message": "An unexpected error occurred.",
    "locations": [{ "line": 1749, "column": 3 }],
    "path": ["saveManyPartNumberPrices"]
  }]
}

We captured 6 consecutive save attempts in our session. All 6 failed with this identical generic error. The user tried varying the shape between attempts (toggling customInputs between null, {DatosPrecio:{Divisa:"USD"}}, and {DatosPrecio:{}}; toggling the line item productId between 14519 and null) — every variation failed the same way.

Root cause (our hypothesis — to be confirmed by your team)

The failed PartNumberDialogProcessQuery in Bug #1 means the RJSF form that builds the price never loaded the catalog of treatment selections, dropdowns, or other process-derived choices. The form therefore submits with treatmentSelections: [], priceTiers: [], priceBuilders: [] even when the underlying processId (215930 or 241753 in our case) has real treatments configured. Server-side, materializing the price record with an empty treatmentSelections for a process that has them triggers an unhandled exception that surfaces as the generic message above.

This is a hypothesis — we cannot see your backend logs. The line/column hint in the error (line: 1749, column: 3) should help your team locate the exact mutation path that throws.

User impact: The generic error message is opaque. Users assume the save worked (HTTP 200) and only later notice their changes were not persisted. We lost roughly two hours of editing time before realizing the saves were silently dropped.
Suggested fix: Two distinct improvements are needed.
  1. Surface the actual exception in the GraphQL error response (or at minimum, a specific error code) instead of the generic "An unexpected error occurred.". Generic messages make support tickets unactionable.
  2. Server-side, validate that partNumberPrices[].treatmentSelections matches the configured process; if it cannot, fail with a specific VALIDATION_ERROR instead of an unhandled exception.
Once Bug #1 is fixed, this downstream symptom should disappear naturally — but the better error surface is still worth implementing for future regressions.

Bug #3 — MEDIUM Archived partNumberSpecs still appear in quote lines

Symptom

When a part number undergoes a spec update via our bulk-upload flow, the previous spec is archived by sending a SavePartNumber mutation with the spec link's id included in partNumberSpecsToArchive. This archives the link correctly: queries that target the part number directly (e.g. GetPartNumber) correctly omit the archived spec.

However, the native quote-line viewer continues to display both specs side by side — the archived (old) one and the active (new) one — as if neither were archived.

Evidence

From a network scan captured earlier today (08:50 local, file scan_results_2026-05-21_085044.json), we observed 5 part numbers (46007-580-01, 46007-902-01, 46008-071-01, 46032-583-01, 48182-577-01) all exhibiting this pattern: their GetQuote-derived shape returns partNumberSpecsByPartNumberId.nodes[] with two entries, one with archivedAt set to a timestamp (the old, archived link) and one with archivedAt: null (the new active link).

Root cause

The query that powers the quote-line viewer (likely GetQuote or one of its companion fragments) selects partNumberSpecsByPartNumberId.nodes without filtering on archivedAt IS NULL. The standalone part-number query (GetPartNumber) filters correctly, so the schema definitely supports the filter — it just needs to be applied here.

User impact: The displayed quote line is misleading: it shows the customer (and our internal users) inspecting the quote that the part number has both spec revisions associated, when in fact only one is active. This causes confusion during quote review, and for a customer-facing PDF it could lead to disputes over which spec applies.
Suggested fix: Add filter: { archivedAt: { isNull: true } } (or your equivalent) to the partNumberSpecsByPartNumberId selection inside GetQuote and any other operation that powers quote-line spec rendering. This mirrors what GetPartNumber already does.

How to reproduce all three on your side

  1. In a domain with several hundred (or more) SpecFieldParam records that have at least one dropdown each (this triggers the pagination limit on allSpecFieldParamDropdowns).
  2. Create or load an existing quote that has a few part-number lines, where at least one part number has a real process with treatment selections defined.
  3. Open the part-number dialog inside the quote for editing → observe in DevTools Network that PartNumberDialogProcessQuery returns HTTP 200 with an error in the response body (Bug #1).
  4. The dialog's treatment / dropdown fields will be empty even though the process has them configured.
  5. Try to save anyway → SaveManyPartNumberPrices returns HTTP 200 with the generic "An unexpected error occurred." (Bug #2).
  6. Separately, take any part number whose latest spec was applied via bulk-upload (replacing a previous spec), and look at the quote line → both spec entries appear, including the archived one (Bug #3).
