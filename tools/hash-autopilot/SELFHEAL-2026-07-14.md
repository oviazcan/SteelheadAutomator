# hash-autopilot self-heal — diagnóstico end-to-end (2026-07-14)

Contexto: rotación de 7 ops (6 queries + `UpdateReceivedOrder`). Corregidas a mano
(scan + deploy 1.7.110). Esta sesión validó **por qué el motor NO las auto-sanó** y
qué falta para "cero captura manual".

## 🎯 HALLAZGO CENTRAL (revisado con evidencia dura): el bloqueador #1 es AUTH, no hidratación
El `route-catalog.json` YA tiene rutas para 6/7 ops (auto-generadas del scan). El motor
las corre. Lo que falla es la **autenticación de la sesión headless**:

- Los tokens ROCP del frontend viven en `Reportes SH/.cache/tokens.json` con
  `expires_at`. **Duran ~8h desde `obtained_at`.** `makeRocpInit` inyecta ese
  `expEpoch` en `localStorage.ROCP_tokenExpire`.
- Cuando el access token venció (comprobado hoy: `expires_at` vencido hace 28 min), el
  SPA lo trata como muerto → intenta refrescar → **el refresh_token también estaba
  revocado** (`steelhead_auth.py` → `400 invalid_grant "refresh token is invalid,
  expired, revoked…"`) → la SPA **redirige a la pantalla de login de Authentik**
  (`/if/flow/steelhead-ui-authentication-flow/`).
- Con la página en el login, **NINGUNA query se dispara** → 0 capturas → el motor
  reporta `authFailed:true`. Eso es lo que pasó en `2026-07-13.json` y `2026-07-14.json`
  (launchd), NO un `TimeoutError` de hidratación como se creyó.

**Prueba directa (misma sesión):** una corrida del motor ~40 min ANTES (token aún
vigente) capturó 24 ops en `home-list`, AllCustomers en `customers-list`, GetDomain en
`domain-config`. La MISMA navegación 40 min DESPUÉS (token vencido) capturó 0 y aterrizó
en `path=/if/flow/steelhead-ui-authentication-flow/`. La única variable que cambió fue
el vencimiento del token.

## Los bloqueadores, en orden de impacto

1. **AUTH headless (EL GORDO, nuevo entendimiento).** Los tokens ROCP vencen ~8h y el
   refresh se revoca → el motor headless cae a login → 0 capturas → `authFailed`.
   - **Inmediato (requiere al usuario):** re-autenticar Steelhead — re-pegar cookie /
     refresh token fresco en `Reportes SH/.env`. Claude NO edita el `.env`.
   - **Diseño (fix del self-healing autónomo):** el wrapper `run-hash-autopilot.sh` debe
     **refrescar el ROCP token ANTES** de abrir el navegador, y **fallar ruidoso** (no
     `authFailed` silencioso) si el refresh no es posible. INVESTIGAR por qué el refresh
     token se revoca: probable **rotación de refresh token** (Authentik emite uno nuevo
     en cada uso e invalida el anterior) — si el idp-flow de `steelhead_auth.py` y el ROCP
     del frontend comparten/pisan el mismo refresh token, o si el nuevo no se persiste, el
     siguiente uso da `invalid_grant`. Sin resolver esto, el motor solo funciona en la
     ventana de ~8h tras un login manual.

2. **Ops que requieren INTERACCIÓN (clic en botón por texto), no navegación.**
   `CreateEditReceivedOrderDialogQuery` ("Editar Orden de Venta") y
   `GetAddPartsReceivedOrder` ("Agregar Piezas (Tabla)") solo se disparan al abrir esos
   modales. El runner soporta `goto` y `clickFirst` (sobre `<a href>`) — **no** clic en
   `<button>` por texto. → Extender `recipe-runner` con paso `clickText` **bilingüe**
   (ES+EN) vía `getByRole('button', {name: /.../})`.

3. **Mutations sin sentinela.** `UpdateReceivedOrder` requiere ejecutarse sobre un objeto
   sentinela (patrón Fase C). No hay entidad declarada en `sentinels-config.json`.

## Qué SÍ está resuelto
- **Detección autónoma** (`probe-config-hashes.mjs`): hoy detectó las 6 queries stale con
  0 falsos positivos. OJO: el probe usa el MISMO token ROCP → también cae si el token
  venció. Hoy corrió mientras estaba vigente.
