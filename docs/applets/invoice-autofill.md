## Lección 0.5.65 (AR matcher reconoce "M.N." = Moneda Nacional, no solo el código ISO)
Bug reportado 2026-07-15: al facturar a **Hubbell** el applet marcaba "sin cuenta AR para MXN" y no resolvía la CXC. Root cause: `findBestARAccount` filtraba las cuentas AR con un **filtro DURO** `/\bMXN\b/i` sobre el `name`. La cuenta de Hubbell se llama `0105-0001-0001-0204 · Hubbell Products Mexico S. de R.L. 1177 M.N.` — el catálogo **no es uniforme**: la mayoría de cuentas traen el código ISO ("... 1140 USD", "... 1200 MXN"), pero otras usan la nomenclatura contable mexicana **"M.N."** (Moneda Nacional = pesos). `/\bMXN\b/` no matchea "M.N." → `byCurrency` vacío → `sin_cuenta_AR_para_MXN`.
- **Fix:** helper `nameMatchesCurrency(name, cur)` reconoce las variantes. **MXN**: `MXN`, `MXP` (ISO viejo), `pesos?`, `moneda nacional`, `nacional`, `M.N.`/`M. N.`/`M.N`, `MN` pegado. **USD**: `USD`, `dólar(es)`/`dolar`, `dollar(s)`, `US$`. Reemplaza el `reCur` duro en el filtro `byCurrency`.
- **Guard anti-falso-positivo:** para "M.N." se exige el **punto** (`\bm\.\s*n\.?`) o la forma pegada `\bmn\b`. Así una razón social con iniciales "M N" sueltas (espacios sin puntos, ej. "Grupo M N Industrial USD") **no** se lee como MXN por accidente. La divisa a resolver ya viene del form/SO/cliente, así que el matcher solo desempata dentro del pool receivable.
- **Gemelo en `bill-autofill` (cuenta AP), portado el mismo día:** su `findBestAPAccount` usa **scoring** (no filtro duro) y ya reconocía `usd/dolar/dollar` y `mxn/peso/nacional`, pero tampoco cubría la abreviatura "M.N." (normaliza a "m n", no a "nacional"). Se le portó el mismo `nameMatchesCurrency` (idéntico) y se cambió el bloque de scoring por divisa a usarlo **pasando el name CRUDO** (`a.name`), no el `nameLower` normalizado — porque la función checa el crudo para "M.N.". Menos crítico ahí (el scoring no elimina la candidata, solo no le suma el +30), pero ahora ambos coinciden.
- **Test:** `tools/test/account-currency-match.test.js` **extrae la función real de AMBOS applets** (conteo de llaves, no copia) y las ejerce con los mismos casos — Hubbell + no-regresión USD/MXN + guard "M N" — más un caso que exige que los dos gemelos coincidan caso por caso (si uno diverge, rojo). 3 suites verdes.

## Auto-fill que reacciona a cambios del usuario (modal manual de Invoice, 0.5.26 → 0.5.34)

Lecciones del ciclo cuando el applet llena varios campos async tras una elección del usuario (cliente seleccionado) y el usuario puede cambiar esa elección a media carrera:

- **Cancellation token, no flag binario.** No basta con `filling = true` para evitar runs paralelos. Si el usuario cambia cliente mientras un run está corriendo, hay que **cancelar** el anterior para que devuelva focus a la UI. Patrón: `runId` monotónico, captura local `myRunId`, `isStale = () => state.runId !== myRunId`, `bailIfStale()` entre cada paso async. Las helpers de retry largo (selectFirstOption, selectOptionMatching) deben aceptar `{ isStale }` y abortar dentro del loop — si no, esperan 4×600ms y siguen abriendo dropdowns aunque el run ya esté muerto. Al cancelar: `body.click()` cierra dropdowns abiertos. (`invoice-autofill.js:1255-1287, 1480-1610`)
- **Idempotencia de acciones "create"**. Si un paso del fill agrega filas/líneas/algo nuevo (ej. clickear "New Line"), agrega un guard que detecte si ya existe antes de re-clickear. Sin esto, cada cambio de cliente apila una entrada vacía. Patrón: `clickNewLine` busca `^Line #\d+$` en el DOM antes de clickear el botón. (`invoice-autofill.js:1469-1488`)
- **Reset `filled=false` al cambiar de input upstream.** Si tienes un guard `if (!state.filled) fill();`, recuerda invalidar `filled` cuando cambia la dependencia (cliente). De lo contrario, después de un fill exitoso, cambiar cliente no re-llena.
- **Network drops se ven como "sin opciones tras retries".** Si los retries de un combobox react-select fallan masivamente y los logs muestran `ERR_HTTP2_PING_FAILED` o `[Network error <Query>]: Failed to fetch`, no es bug del applet — es que la query GraphQL del propio Steelhead que pobla las opciones no respondió. No subas retries indefinidamente; mejor reportar `success:false` y dejar que el usuario reintente (la cancelación lo permite).
- **Pausar el fill mientras el usuario interactúa con el upstream input.** El `singleValue` de un react-select solo cambia cuando el usuario *confirma* una opción, así que el listener "cliente cambió → cancela y rellena" se dispara después del select. Pero el usuario puede *re-abrir* el combobox (sin cambiar singleValue todavía) para escribir un cliente diferente — durante ese tiempo el fill sigue corriendo y le roba focus al usuario. Patrón: (1) `isCustomerComboboxActive()` mira `document.activeElement` dentro del wrapper del combo + `aria-expanded="true"`; (2) `focusin` listener global con capture: si el target cae dentro del combo → bumpa `runId` y marca `state.deferred = true`; (3) `isStale` del fillImpl considera `deferred` como stale; (4) `handleManualModal` no relanza fill mientras `isCustomerComboboxActive()` — solo lo hace cuando el observer ve que el usuario salió y `deferred` sigue true. Sin esto, cambiar de cliente a media carrera dispara dropdowns paralelos (Bill To/Ship To/Contact) que pelean por focus con el combobox de Customer. (`invoice-autofill.js:1277-1302, 1252-1318` en 0.5.33)
- **Lectura de un campo específico: label-driven, NO walking-up desde singleValues.** Buscar `singleValue`s en todo el documento y caminar hacia arriba buscando un sibling con cierto texto suena robusto pero falla en layouts donde varios labels (`<p>Customer:</p>`, `<p>Terms:</p>`, ...) son siblings dentro del **mismo** wrapper (el modal manual de Invoice usa `css-iyrxkt` así). Al limpiar Customer su singleValue desaparece y el walker, partiendo del singleValue de Terms, llega al ancestro común y encuentra el `<p>Customer:</p>` de OTRO field como sibling — devuelve el value de Terms (ej. `"10 Días"`) como si fuera el cliente, disparando ciclos espurios de "customer changed → fill". El fix correcto es localizar el wrapper exacto del campo por su label (`findReactSelectControlByLabel`) y consultar el singleValue **solo dentro** de ese container. Sin singleValue ahí → `return null`. Aplica también a forms RJSF y a cualquier otro caso donde lees el "valor actual" de un combobox específico. (`invoice-autofill.js:723-783` en 0.5.34)

## Lecciones 0.5.60 → 0.5.63 (PS-embedded, AR matcher)
Cuando el usuario abre una invoice desde `Domains/<id>/Shipping/PackingSlips/<id>` (en lugar del flujo "Create Invoice Manually"), Steelhead **renderiza el editor de invoice in-place sin cambiar la URL** y con un layout DOM **distinto** al modal manual. Lecciones del ciclo:

- **URL gate: `/Shipping/PackingSlips` también activa el applet.** El gate inicial solo matcheaba `/Domains/<id>/Invoices`, así que el panel no aparecía en el flujo PS aunque el editor estuviera presente. Patrón: `INVOICE_URL_RE = /\/Domains\/\d+\/(?:Invoices|Shipping\/PackingSlips)(?:\/|$)/`. Aplica a otros applets que se inyectan por URL: si el feature tiene varios "modos" de entrada, lista todos los paths que llevan al mismo editor antes de hardcodear uno solo. (`invoice-autofill.js:16` en 0.5.60)
- **Layout de líneas en flujo PS: dos sub-tables, no form-card con `<p>INCOME</p>`.** Cada línea vive en `<div class="css-7a0s5b">` con (1) header table de 3 columnas (`<th>Line #N - PN</th>`, `<th>Description:</th>`, `<th>Total: $X</th>`) y (2) data table de 11 columnas (Include, Product, Línea, Departamento, Description, Quantity, Price, Subtotal, Tax Code, Close SO Line, **Income Account**). El extractor original buscaba un `<p>INCOME</p>` italic adyacente al combobox (que solo existe en el modal manual / form-card). En PS-embedded ese subtítulo no existe; "Income Account" solo aparece como **encabezado de columna**. Patrón robusto: (1) localizar `<th>Line #N - PN</th>`; (2) walk-up al wrapper que contenga otra table cuyo `:scope > thead > tr > th` incluya `^income account$`; (3) tomar el data row del `<tbody>` con `cells.length === columnHeaders.length` (filtra el sub-row con `colspan=11` de Part Numbers/Locations); (4) `incomeCell = dataRow.cells[incomeIdx]`; (5) **anclar `incomeLabel = incomeCell.firstElementChild`** para que `tryFillIncomeInLine` haga `host = incomeLabel.parentElement = <td>` y `host.querySelector('input[role="combobox"]')` quede scopeado SOLO al td de Income — si anclas en el `<td>` directo, `host = <tr>` y captura el primer combobox del row (Product), no el de Income. Mantener layout legacy como fallback (`if (lines.length > 0) return lines;` antes de barrer headings con `<p>INCOME</p>`). (`invoice-autofill.js:918-1015` en 0.5.63)
- **AR matcher con muchas candidatas: tres defensas combinadas.** Cuando el AR account list trae 60+ opciones para un cliente USD (Schneider Electric MX), tres bugs convergen: (1) regex de currency suffix `\b<CUR>\s*$` excluye accounts donde el sufijo es token interno (`"... 1128 USD"` por truncado UI), debe ser `\b<CUR>\b`; (2) `customer.name` puede venir vacío en el shape — caer a `state.customerName` para construir tokens del cliente; (3) si tras filtros sigue habiendo `>3` candidatas, marcar `tooMany: true` y devolver `account: null` con `reason: 'demasiadas_candidatas_<N>'` — mejor decirle al usuario "no supe cuál" que adivinar. El panel renderiza ✗ con tooltip explicativo en ese caso. (`invoice-autofill.js:576-650` en 0.5.60-0.5.62)
- **Auto-collapse del panel cuando todo está OK.** Patrón: tras `renderPanel()`, `setTimeout(() => { if (isAllDone()) collapse(); }, 1800)`. `isAllDone()` chequea AR + cada línea con resultado success-truthy. Si el usuario cambia algo después, el rerun reabre el panel naturalmente.
- **Re-confirmación de la regla del wrapper HTML.** El ciclo 0.5.60→0.5.63 cerró en una iteración después de pedir el HTML del wrapper; antes hubo dos rondas adivinando que perdimos. **No itera deploys para descubrir el layout** — pide el wrapper, escribe el extractor una sola vez. Esta es la misma regla que ya está en "Trabajo con UI / DOM de Steelhead" arriba; el ciclo simplemente la confirma de nuevo.

## Lecciones 0.5.64 (modal OV/PS rediseñado: AR Account label, income commit, gate del panel)
Steelhead rediseñó el editor de invoice embebido (flujo OV/Packing Slip). Tres regresiones, todas diagnosticadas en una sola pasada con el outerHTML del wrapper + un inventario de comboboxes por consola:

- **La Cuenta CXC (AR) aterrizaba en "Ship Via".** El modal nuevo reemplazó el subtítulo italic `<p>ACCOUNTS_RECEIVABLE</p>` por un label plano `<p>AR Account:</p>`, y estructuró el campo así: dentro de un `css-iyrxkt`, el orden de hermanos es `… <div>Ship Via (react-select-24)</div> … <p>AR Account:</p> … <div style="width:100%">react-select-25</div>`. El label y su combobox **NO comparten un wrapper exclusivo** (son hermanos sueltos del mismo contenedor). El viejo `tryFillCombobox` subía del label al ancestro común y hacía `querySelector('input[role="combobox"]')`, que devuelve el PRIMER combobox del bloque en orden de documento = **Ship Via**, no AR. Fix: anclar por label con `tryFillReactSelectByLabel(AR_LABEL_RE, …)` → `findReactSelectControlByLabel` → `findFieldContainerByPLabel`, que camina del `<p>AR Account:</p>` al **sibling siguiente** (el `<div style="width:100%">` con react-select-25). Se quitó del flujo de AR el `tryFillCombobox` peligroso; el único fallback que queda es el subtítulo legacy `<p>ACCOUNTS_RECEIVABLE</p>`, que es fail-safe (si no existe el subtítulo, `tryFillARBySubtitle` retorna `success:false` sin tocar ningún combobox). `AR_LABEL_RE = /^\s*ar\s*account\s*:?\s*$|^\s*cuenta\s*(?:por\s*)?cobrar\s*:?\s*$|^\s*accounts?\s*receivable\s*:?\s*$/i`. (`invoice-autofill.js` fillAllFields)
- **La cuenta de Income por línea quedaba "a medio seleccionar".** `clickAndSelectOption`→`pickBestOption` confirmaba la opción con `best.click()`. react-select v5 selecciona la Option en **mousedown** (hace `onMouseDown→preventDefault` para no perder focus); un `.click()` sintético solo no dispara el commit. Síntoma: el accountNumber tecleado persiste en el input y el menú queda abierto, **sin `singleValue`** (visible en el inventario por consola: `valor="0401-0004-0000-0000"` sin singleValue confirmado). Fix: disparar la secuencia `mousedown→mouseup→click` sobre la opción. Beneficia a AR y a income (ambos pasan por `pickBestOption`). Las helpers del modal manual (`selectFirstOption`/`selectOptionMatching`) ya usaban mousedown vía `openSelectAndGetOptions`, así que no se tocaron.
- **El panel se plantaba sobre la vista de detalle de un Packing Slip (dos rondas).** `scanForInvoicePage` daba `found=true` por `rjsfInputs.length >= 5` (cualquier form RJSF). La página `/Domains/N/Shipping/PackingSlips/<id>` y el dashboard `/Domains/N/Shipping` (detalle/lista de albaranes, **sin** editor de invoice montado) caen dentro del URL gate — que incluye `/Shipping/PackingSlips` porque la factura se crea in-place desde ahí sin cambiar URL — y traen su propio form RJSF con ~87 inputs `root_*` → panel intrusivo "Esperando selección de cliente…" sobre el albarán.
  - **Ronda 1 (insuficiente):** reemplazar `rjsfInputs.length >= 5` por `root_DatosContables_*` (genérico) o heading. **No bastó:** la vista del albarán **también** renderiza el bloque de datos contables del **cliente** (`root_DatosContables`, `root_DatosContables_DivisaUSD`, `_DivisaMXN`, `_CuentasContables_*`, `_EmpresaEmisora`…), así que `[id^="root_DatosContables"]` seguía dando `true`. Diagnóstico clave: snippet que evalúa cada término de `found` → en el albarán `divisaInput:false, tcInput:false, datosContablesAny:true, invoiceForHeadings:[]`.
  - **Ronda 2 (correcta):** `const found = !!(divisaInput || tcInput || hasInvoiceHeading)`, donde `divisaInput = root_DatosContables_Divisa` y `tcInput = root_DatosContables_exchangeRate` son **exclusivos del editor de factura** (¡distintos de `root_DatosContables_DivisaUSD`/`_DivisaMXN`, que son flags del cliente y SÍ están en el albarán!), y `hasInvoiceHeading` matchea `/(?:creating|editing|new|create|edit)\s+invoice\s+for\b/i`. El albarán no tiene ninguna de las tres → no aparece. La factura OV/PS monta el heading "Creating Invoice for X" → sí. La factura manual no depende de `found` (cae a `isManualInvoiceModal`). **Lección:** `root_DatosContables` (genérico) NO es señal de "editor de factura" — el cliente trae su propio bloque de datos contables en otras vistas; ancla en los campos hoja específicos de la factura.
- **"No resuelto" espurio en la cuenta de ventas por race de carga del catálogo.** Con clientes 'general' (prefijo `0401-0001`), la primera pasada de `runAutofill` a veces corría con `state.allAccounts` parcial — tenía las receivable `0105-*` (que resolvieron la CXC) pero aún no las de ventas `0401-*`, porque `InvoiceLowCodeData` llega async **después** del primer run. `resolveLineAccount` marcaba "No resuelto (0401-0001)" y sólo se corregía con un re-run manual (↻ actualizar). Fix (lógica pura, sin DOM): `hasSalesAccounts(arr)` detecta si el catálogo trae alguna `/^040[12]/`; `ensureSalesAccountsLoaded()` reintenta cargar (SearchAccounts `'%%'` + da chance a que los interceptores pueblen `state.allAccounts` entre reintentos, 5×500ms) **antes** de resolver líneas, sólo si faltan ventas (no-op si ya están). Además `resolveLineAccount` marca `catalogIncomplete` cuando el catálogo no trae ninguna `040[12]-*` → el panel muestra "Cargando catálogo de cuentas…" (status `pending`) en vez de "No resuelto" (status `warn`), distinguiendo "aún cargando" de "la cuenta no existe de verdad". `isAllDone()` sigue devolviendo false mientras una línea no tenga `account`, así que el panel no auto-colapsa hasta resolver. (`invoice-autofill.js` ensureSalesAccountsLoaded / _runAutofillInner / resolveLineAccount)
- **Re-confirmación (otra vez más) de la regla del wrapper HTML.** El `<p>AR Account:</p>` hermano de Ship Via era invisible sin el DOM real; el inventario de comboboxes por consola (`SUBTITULOS_MAYUS` + label/valor por combo) fue clave para confirmar el "a medio seleccionar" sin adivinar. Cero iteraciones de deploy para descubrir layout.
