Steelhead · Power Tools · Feature Request

Expose the outbound packing slip (remisión) to the Invoicing low-code hook

Hook: getInvoicePricing Area: Power Tools — Invoicing Date: 2026-06-04 Submitted by: Ecoplating

Summary

The Power Tools Invoicing hook (getInvoicePricing, which produces lowCodeDefaultInvoiceLineItems) cannot read the outbound packing slip / shipment ("remisión") for the parts being invoiced. We'd like that object exposed in the hook's Input so we can validate it and write it onto each invoice line ourselves, before the invoice is saved.

Context

  • We use getInvoicePricing to build CFDI (Mexican tax) invoice line descriptions. The description field is tightly space-constrained and its exact content matters for the customer and the SAT integrator.
  • We ship parts before invoicing — the parts are already at the Embarcando recipe node when the invoice is built — so the outbound packing slip already exists at invoice time.
  • Today, after the invoice is saved, Steelhead automatically appends the outbound packing slip number to each line's description (e.g. , Packing Slip: 1469). That auto-append is configurable and we can turn it off — but the hook runs before save and never sees the packing slip value, so disabling the auto-append would make the remisión disappear entirely (the hook has nothing to write in its place). That is the blocker: we can't replace the raw append with a validated, properly formatted value because the value isn't exposed to the hook.

What's currently exposed (evidence)

A dump of a partAccount from the getInvoicePricing Input shows these keys only:

id, certReport, station, locationId, locationPath, quantity,
recipeNode, partGroup, rack, receivedBatch, measurement

The only "packing slip"-like field is receivedBatch.customInputs.DatosRecibo.PackingSlip — that is the inbound (receiving) PS entered by the customer, not the outbound shipment. There is no outbound packingSlip object anywhere in this Input.

FieldMeaningIn getInvoicePricing?
receivedBatch…DatosRecibo.PackingSlip Inbound PS — entered at receiving (customer's slip) Yes
partAccounts[].packingSlip.idInDomain Outbound shipment / remisión — what we need No

The model already supports it

The Invoice PDF hook (getPdfCustomization / INVOICE_TEMPLATE) already exposes the outbound shipment on its part accounts:

lineItem.referencedPartAccounts[].packingSlip {
  idInDomain, createdAt, shipVia, shipMethod, trackingNumber, shipToAddress
}

So the data is in the graph and already resolved for a sibling hook — this is purely a matter of the getInvoicePricing query requesting it.

Request

Request

Add the outbound packing slip to the part accounts already returned by the getInvoicePricing query — minimally:

partAccounts {
  ...existing fields...
  packingSlip { idInDomain }   // outbound shipment / remisión
}

With the field exposed, the hook can:

  • Validate before save — surface a pre-save message to the operator (e.g. "This invoice will carry remisión(es): 1469, 1470 — confirm before saving"), catching a wrong or missing remisión before the invoice is committed.
  • Write the remisión itself — with the correct format and placement — after we disable Steelhead's auto-append (already configurable on our side). The auto-append exists today only because the hook has no way to add the remisión; exposing the field removes that dependency, with no risk of a duplicate.

Why it matters

CFDI descriptions are length-limited and customer-sensitive. Exposing the packing slip lets us validate the remisión before save (preventing bad invoices) and write it into the description ourselves — within SAT limits and consistent with the PDF — instead of relying on a raw, uncontrolled append.
