Thank you for flagging the performance concern. We take it seriously and want to be fully transparent about what our internal browser extension does, how it talks to Steelhead, and the change we shipped today to reduce its request volume. We also include one observation that may help on the platform side. Our goal is a tool that stays well within reasonable usage and is a good citizen on shared infrastructure.

1. What the extension is

It is an internal, client-side Chrome extension used only by Ecoplating staff to automate repetitive data-entry tasks in the Steelhead UI (bulk quote/part-number entry, invoice auto-fill, line routing, etc.). Key facts:

  • No backend of our own. It runs entirely in the operator's browser tab.
  • It uses the same public GraphQL endpoint and persisted queries that the Steelhead web app itself uses (POST /graphql, Apollo persisted-query SHA-256 hashes, credentials: include). It does not use any private or undocumented API.
  • It acts strictly as the logged-in user, with that user's own session cookies and permissions. It cannot see or touch anything the operator could not already do by hand in the UI.
  • No data leaves the browser; nothing is scraped to an external server.

2. Why it is designed this way

The tasks it automates are otherwise done by hand, click-by-click, in the native UI — slow and error-prone at our volume. The extension performs the same operations a human would, just faster and with fewer mistakes. In several cases it actually issues fewer requests than the manual flow it replaces:

TaskManual flow (native UI)Extension
Re-routing a work order between lines ~33 react-select dropdowns, each tied to its own lookups; ~17 min/order One batch mutation (CreateUpdateDeleteRoutes) after computing the mapping locally
Candidate stations per treatment Up to 17+ SearchStationsForTreatment calls per order Zero extra calls — reuses the candidates already embedded in the routing tree response
Bulk price / part-number upload One round-trip per line, manually Batched (e.g. 20 PNs per SaveManyPartNumberPrices)

So the design intent is explicitly to lower total request count versus manual operation, not raise it.

3. The issue we found, and the fix we shipped today

We audited the extension after your message and identified one recently-added feature whose request volume was higher than it should be: a hover tooltip on the Scheduling board that shows a part's base metal and customer packing slip.

  • Before: it pre-fetched that data for every part row as it scrolled into view. On a large board this could fan out to thousands of small GetPartNumber / GetInventoryBatch reads triggered passively by scrolling.
  • After (deployed 2026-06-24): the tooltip is now strictly on-demand — it issues a read only when the operator actually hovers a specific part, and caches the result. Request volume is now proportional to what a person genuinely inspects (a handful per session), with no scroll-driven traffic.
Result: the passive, scroll-driven request pattern is eliminated. We also removed an automatic "route the whole board" path that could load many work-order trees at once; routing now requires an explicit selection.

4. One observation that may help on the platform side

While profiling, we noticed the board's own native query RelatedSchedulingInformation is very heavy: a single load returns on the order of ~87 MB, dominated by allWorkOrders returning every work order in the domain un-paginated (~1,700+ rows), each eager-loading ~10 nested relations. This is native Steelhead behavior, independent of our extension, but it may be a meaningful contributor to perceived load on the Scheduling board. Paginating it and lazy-loading the heavy relations would likely help all tenants. We're happy to share the captured request/response shapes if useful.

5. Our commitments

  • Keep request volume proportional to real operator work; no passive/background fan-out.
  • Batch writes where the API supports it; reuse embedded data instead of re-querying.
  • Honor any rate limits, headers, or usage guidance you provide. If you can point us to recommended limits or a preferred pattern for high-volume operations, we'll adopt them.
  • Share our network captures (exact operations, hashes, variable shapes) on request so you can verify the traffic.

We want this to work well for everyone on the platform. Please tell us if you'd like specific changes or limits on our side, and we'll implement them promptly.
