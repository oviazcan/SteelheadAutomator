Steelhead ERP · Bug Report & Resolution

Certificate of Conformance PDF renders blank when a spec parameter has no recorded measurement

Template: pdfType = CERTIFICATION_TEMPLATE (pdfTemplateId 1294239) Domain: Ecoplating (TLC) Date: 2026-05-29 Status: Worked around client-side

1. Summary

Some Certificates of Conformance (CoC) were generated blank: the PDF kept the header and footer but the entire body (Related Documents, Production Data, and the measurements table) was missing. We traced this to the PDFGeneratorAPI certification template: it does not guard against a null value on spec parameters that were defined but never measured. Accessing a property of that null throws during render and drops the whole per-part body block. We deployed a workaround in the low-code getPdfCustomization hook, but the underlying fragility lives in the template and would be best fixed on the Steelhead side.

2. Symptom

Operators reported that certificates #38 and #39 printed blank, while #35 and #37 printed correctly. "Blank" did not mean a zero-byte file — the PDFs were 200–280 KB and rendered the company header, customer block, the certification statement, and the "Certified by" footer. What was missing was the body:

  • Related Documents (PO, Work Order, Lot)
  • Production Data (Part Number, Quantity, Weight)
  • The measurements table (attribute / unit / min / max / measurement / inspector)

3. Investigation

We ruled out the obvious suspects with direct evidence:

  • Not an empty parts list. Cert #38 genuinely has 0 associated parts (an expected empty case). But cert #39 had 9 measurements linked to parts transfer accounts and still rendered blank.
  • Not the low-code hook. The hook produced a valid additionalPayload for cert #39: a one-element partsTransferAccounts with quantityWithConversion = "Cantidad: 550 Peso: 16.56 KGM". The hook ran without throwing.
  • The differentiator. A structural diff of cert #37 (renders) vs cert #39 (blank) surfaced one decisive divergence: in cert #39, several specs[].specFields[].valuesAndParams[] entries had value: null — parameters that were defined but never measured. Cert #37 had none.

4. Root Cause

The certification template iterates specs[].specFields[].valuesAndParams[] and reads fields off each entry's value object (e.g. value.recordedValueAsString, value.passed). For a parameter that was defined but never measured, value is null. Reading a property of null throws during rendering, and because the entire per-part body section is built inside that same iteration, the whole body block fails to render. The header and footer live outside that block, which is why they still appear.

In other words: a single unmeasured parameter is enough to blank the entire body of an otherwise valid certificate.

5. Our workaround (low-code hook)

Because we cannot edit the PDFGeneratorAPI template directly, we hardened the low-code getPdfCustomization hook (pdfType = CERTIFICATION_TEMPLATE) to strip unmeasured entries before they reach the template: 

// Drop measurement entries that were never recorded (value === null),
// which otherwise blank the entire certificate body.
const hasValue = (entry) => !!entry && entry.value != null;

partsTransferAccounts.forEach((pta) => {
  (pta.specs ?? []).forEach((spec) =>
    (spec.specFields ?? []).forEach((sf) => {
      sf.valuesAndParams      = (sf.valuesAndParams ?? []).filter(hasValue);
      sf.couponValueAndParams = (sf.couponValueAndParams ?? []).filter(hasValue);
    }));
  (pta.partAccountAncestors ?? []).forEach((anc) => {
    anc.specValuePartsTransferAccounts =
      (anc.specValuePartsTransferAccounts ?? []).filter(hasValue);
  });
});

Trade-off: unmeasured parameters no longer print on the certificate (same behavior as a fully-measured certificate, which only shows recorded rows). Nothing is fabricated — we simply stop the document from blanking entirely.

Deployed as the active CERTIFICATION_TEMPLATE low-code version id = 10542 and verified against the real cert #39 payload.

6. Recommendation for Steelhead

The durable fix belongs in the PDFGeneratorAPI certification template: null-guard every access to a parameter's value (e.g. only render the measurement cell when value is present, otherwise leave it empty). That way a parameter defined but not yet measured renders as a blank cell instead of crashing the whole document body.

This would also help any other tenant whose certificates include not-yet-measured spec parameters, and would let unmeasured rows still appear on the CoC (which our hook-side workaround currently drops).

7. Reproduction notes

  1. Create a CoC whose spec has at least one parameter with no recorded measurement (value: null).
  2. Generate the certificate PDF.
  3. Observed: header + footer render; the body (Related Documents, Production Data, measurements table) is missing.
  4. Expected: body renders, with the unmeasured parameter shown as an empty/blank measurement cell.
