Los comandos concretos envejecen; el repo no

Esta guía cita rutas, flags y versiones exactas del repo al corte de esta redacción (2026-07-23). Eso es intencional — es lo que hace la guía accionable — pero también significa que puede quedarse atrás cuando el código cambie. En una frase

Clonas el repo, corres tools/install-hooks.sh una vez, editas JavaScript vanilla en remote/scripts/, corres tools/run-tests.sh antes de tocar nada más, y cuando quieres que un cambio llegue a los operadores corres tools/deploy.sh (o tools/wb-deploy.sh si trabajas en un worktree) — que bumpea versión, firma con KMS, espeja a gh-pages y publica. Nada de lo que edites en main afecta a nadie hasta ese paso.

01Panorama de la arquitectura para desarrollar

El detalle conceptual completo vive en el Manual del Administrador y en la Ficha profunda: Hash-Autopilot. Aquí solo lo que necesitas para orientarte antes de tocar código.

El sistema tiene dos mitades con ciclos de vida distintos:

  • extension/ — el cascarón: una extensión Chrome MV3 empaquetada (manifest, background.js, content.js, popup.html/js). Se instala una vez y casi no cambia.
  • remote/ — el cerebro: los scripts de cada applet (remote/scripts/*.js) y el catálogo de configuración (remote/config.json, hashes de persisted queries + versión). Se publican en la rama gh-pages (servida por GitHub Pages) y el cascarón los descarga en tiempo de ejecución.

El hallazgo que cambia cómo pruebas localmente

extension/background.js tiene hardcodeada la URL de origen de los scripts remotos: const REMOTE_BASE_URL = 'https://oviazcan.github.io/SteelheadAutomator'. Esto significa que aunque cargues la extensión "descomprimida" desde tu clon local (chrome://extensions → Cargar descomprimida → carpeta extension/), el applet que se ejecuta en el navegador sigue siendo el que vive en gh-pages — NO tu copia local de remote/. Editar remote/scripts/foo.js en tu clon no cambia lo que ve el navegador hasta que lo despliegues.

No hay un comando tipo npm run dev con recarga en caliente contra tu copia local. El flujo real de prueba es: (a) cubrir la lógica con el golden test puro (sección 5) — la vía principal —, o (b) desplegar de verdad a gh-pages y probar en vivo. Hay un truco documentado en planes viejos (docs/superpowers/plans/2026-05-07-receiver-date-override.md, 2026-05-18-process-duplicates.md): levantar python3 -m http.server sirviendo remote/ localmente, apuntar temporalmente REMOTE_BASE_URL a http://localhost:<puerto> y agregar ese origen a host_permissions de manifest.json — pero no es un flujo con script propio en tools/, y hay que revertirlo antes de comitear. ⚠️ confirmar con el equipo si existe/conviene formalizar un modo "dev server".

Reglas de desarrollo que aplican a todo remote/scripts/*.js (detalladas en CLAUDE.md raíz):

  • JavaScript vanilla — sin React, sin frameworks, sin bundlers (nada de npm run build del lado de la extensión).
  • Código y variables en inglés; documentación y UI en español.
  • UI propia siempre en dark mode (#1c2430 base, #e6e9ee texto, acento #13a36f) — para que el operador distinga de un vistazo la UI de la extensión de las pantallas nativas de Steelhead, que son claras.
  • Núcleo puro + golden test. La lógica de negocio de cada applet se separa en un módulo *-core.js / *-engine.js sin DOM ni red, con su archivo de test en tools/test/. Ver sección 5.
  • Constantes de dominio en config.json, no hardcodeadas en el script.

03Obtener el código

git clone https://github.com/oviazcan/SteelheadAutomator.git
cd SteelheadAutomator

Para desarrollo de applets normal, esto es todo lo que necesitas clonar. Pero si vas a mantener el hash-autopilot (sección 10) — que sincroniza hashes de persisted queries con otros dos proyectos — necesitas clonar también los repos hermanos, en la disposición exacta de carpetas que el código espera (varios scripts tienen la ruta hardcodeada, no la derivan de tu $HOME):

~/Projects/Ecoplating/
├── SteelheadAutomator/     (este repo)
├── Reportes SH/            (nota: el nombre trae un ESPACIO — entrecomilla en shell)
└── SteelheadPowerTools/    (github.com/oviazcan/SteelheadPowerTools)

Rutas absolutas hardcodeadas a una sola cuenta — ⚠️ confirmar con el equipo

Varios archivos del hash-autopilot no derivan la ruta de los repos hermanos desde $HOME/os.homedir() — la tienen escrita literal como /Users/oviazcan/Projects/Ecoplating/…:

  • tools/hash-autopilot/hash-autopilot.mjs (línea ~100, TOKENS_PATH) — de ahí lee los tokens de sesión para autenticarse.
  • tools/hash-autopilot/external-sinks.json — los tres destinos externos donde escribe hashes rotados.
  • tools/hash_sources.py — REPORTES_SH_CLIENT y POWERTOOLS_ROOT.
  • Los .plist de tools/launchd/ que agendan la corrida horaria (WorkingDirectory y ProgramArguments).

En la práctica, el hash-autopilot agendado (cron) hoy vive en una sola Mac dedicada ligada a esa cuenta. Si otra persona va a correrlo (aunque sea en modo --dry-run) en otra máquina — con otro nombre de usuario, o en Windows/Linux —, tiene que editar esas rutas primero o replicar exactamente /Users/oviazcan/… en su sistema. No es un bug pequeño de "olvidé generalizar": es una decisión implícita de que esta pieza corre en una máquina fija. Pregúntale al equipo cuál es el plan si esa máquina cambia.

04Instalación

1. Instala los git hooks (obligatorio, una vez por clon)

tools/install-hooks.sh

Copia todo lo que hay en .githooks/ al directorio de hooks común del repo (git rev-parse --git-common-dir), así que aplica a todos los worktrees que crees después, sin volver a correrlo — excepto si editas un hook, ahí sí hay que re-instalarlo. Hoy solo hay un hook: pre-push, que al pushear gh-pages bloquea el push si (a) no espeja main:remote/ byte a byte, (b) config.json version no sube respecto a lo publicado, o (c) config.sig no verifica contra la llave pública embebida. Sin este hook instalado, un git push origin gh-pages a mano podría dejar la extensión de los operadores rota o desincronizada sin que nadie lo note hasta que alguien reporte un applet caído.

2. Dependencias Node (solo dentro de tools/hash-autopilot/)

La raíz del repo no tiene package.json — no hay npm install a nivel repo. El único proyecto Node con dependencias declaradas es el hash-autopilot:

cd tools/hash-autopilot
npm install
npm run install-browser   # playwright install chromium — solo si vas a correrlo

Instala solo si vas a trabajar en la sección 10. Para editar applets y correr la suite de tests no lo necesitas.

3. Dependencias Python (ad hoc, sin manifiesto)

Tampoco hay requirements.txt ni pyproject.toml. Los paquetes de terceros que de verdad usan los scripts de tools/*.py son, verificado por grep de imports:

pip install pytest openpyxl

pytest corre los tests tools/test_*.py; openpyxl lo usan los scripts que leen/escriben .xlsx (catálogos, snapshots). El propio CI instala exactamente estos dos (python -m pip install --quiet pytest openpyxl).

Sin pytest, la suite se salta la mitad en silencio

Verificado en vivo: si pytest no está instalado, tools/run-tests.sh imprime "(pytest no disponible — omito Python…)" y aun así reporta "suite VERDE" — porque el veredicto solo cuenta lo que corrió. No confundas "verde" con "toda la suite pasó"; confirma que ves la línea ▶ Python (pytest) corriendo casos, no solo el aviso de que se omitió.

4. Solo si vas a desplegar: gcloud + acceso a KMS

Ver sección 9 completa — no es opcional si tu trabajo incluye publicar cambios a gh-pages.

05Correr las pruebas

tools/run-tests.sh              # corre todo, resumen
tools/run-tests.sh -v            # salida completa de cada archivo
tools/run-tests.sh --no-python   # solo JS, si no tienes pytest a la mano

Corrí la suite ahora mismo desde este worktree para dar una cifra real, no una de bitácora vieja: 71 archivos JS, 0 rojos (tools/test/*.test.js + los legacy tools/test_*.js de la raíz), Python omitido en este entorno por falta de pytest. Esa cifra sube con el tiempo — no la tomes como un número fijo, solo como evidencia de que el comando funciona y de qué se ve un run verde.

Qué cubre: los golden tests de los módulos puros de cada applet (*-core.js, *-engine.js, *-classify.js, etc.) — la lógica de negocio aislada de DOM y de red. Cada .test.js se corre aislado vía node --test <archivo> (no node --test <directorio>, que según el comentario del propio script no resuelve el directorio en Node 25) porque algunos applets cargan el IIFE completo en un sandbox vm y correrlos juntos compartiría estado de módulos entre tests.

Esta suite es el gate de calidad del deploy: tools/deploy.sh y tools/wb-deploy.sh la corren automáticamente antes de tocar producción y abortan si sale roja (bypass de emergencia: SH_SKIP_TESTS=1, úsalo sabiendo por qué). También corre en CI (.github/workflows/tests.yml) en cada push/PR que toque remote/scripts/**, tools/** o .github/workflows/** — con Node 22 y Python 3.9, e instala pytest+openpyxl antes.

06El ciclo de desarrollo

  1. 1

    Editas

    Casi siempre en remote/scripts/<applet>.js (+ su módulo puro *-core.js si aplica) y, si agregas un applet o un hash nuevo, en remote/config.json. Editar en main (o en cualquier rama/worktree) no afecta a ningún operador hasta que despliegues — ver la advertencia de la sección 1 sobre por qué "cargar descomprimida" tampoco basta para ver tu cambio en vivo.

  2. 2

    Escribes/actualizas el golden test

    Si tocaste lógica de negocio, el módulo puro correspondiente en tools/test/<applet>-core.test.js (o el que aplique) debe reflejarlo. La skill nuevo-applet trae el scaffold de "core puro + golden test + bitácora" para un applet desde cero.

  3. 3

    Corres la suite

    tools/run-tests.sh. Si sale roja, no sigas — arréglala antes de desplegar (el deploy la vuelve a correr y aborta igual).

  4. 4

    Pruebas en el navegador (si aplica)

    Chrome → chrome://extensions → activa "Modo de desarrollador" → "Cargar descomprimida" → selecciona la carpeta extension/. Confirma en el popup que dice "Conectado". Recuerda: para ver un cambio de remote/scripts/ reflejado ahí, primero tiene que estar publicado en gh-pages (sección 8) — o usar el workaround del servidor local de la sección 1.

  5. 5

    Despliegas

    tools/deploy.sh (desde el worktree de main) o tools/wb-deploy.sh (desde otro worktree). Ver sección 8.

  6. 6

    Actualizas la bitácora

    Cada applet tiene su bitácora en docs/applets/<applet>.md, indexada desde CLAUDE.md. La skill wrap ayuda a reconciliarla al cerrar una sesión de trabajo.

Nota Windows — descarga de zips y antivirus

INSTALAR.md (la guía para el usuario final) documenta que McAfee en Windows a veces renombra .js a .j_ al descomprimir el zip distribuido de la extensión, y hay que quitarles el guion bajo a mano. Esto es sobre el zip empaquetado que se distribuye a operadores, no sobre un git clone — pero si en algún punto empaquetas y pruebas el .zip en una máquina Windows con McAfee, es el primer sospechoso si los scripts "desaparecen".

Construir el .zip distribuible — ⚠️ confirmar con el equipo

No encontré un script en tools/ que empaquete extension/ en el steelhead-automator.zip que se distribuye (mencionado en INSTALAR.md y publicado en gh-pages). El proceso de empaquetado parece manual. Antes de que te toque republicar el .zip (solo aplica cuando cambia código de extension/, no de remote/), confirma con el equipo el procedimiento exacto.

07Trabajo con worktrees (trabajo paralelo)

Para correr dos sesiones de desarrollo (dos personas, o dos hilos de Claude Code) sobre el mismo repo sin pisarse, el repo usa git worktrees — cada uno es un directorio aislado en su propia rama, compartiendo el mismo .git.

tools/new-worktree.sh <feature-name> [branch-base]
# ejemplo:
tools/new-worktree.sh dup-validator-tier4
# resultado: ../SteelheadAutomator-dup-validator-tier4 en rama wt/dup-validator-tier4

Este mismo documento vive en un worktree llamado workbench (rama workbench, checkout en .claude/worktrees/workbench/ del repo principal) — un banco de trabajo persistente separado de main, con sus propias reglas locales en CLAUDE.local.md (no versionado).

Hot filePor qué duele editarlo en paralelo
remote/config.jsonBump de versión + hashes compartidos — casi seguro choca si dos sesiones lo tocan a la vez.
CLAUDE.mdÍndice de applets + reglas globales — mismo problema.
rama gh-pagesEs el espejo de deploy — solo una sesión debe estar deployando a la vez.

Si deployas DESDE un worktree que no es main

No lleves el script a mano con git show workbench:remote/scripts/foo.js > … + tools/deploy.sh. deploy.sh hace git add remote/ en el worktree de main, así que arrastraría dentro de tu commit de deploy cualquier WIP sin commitear que otra sesión tenga ahí (pasó de verdad: casi mezcla el WIP de otra feature con un deploy no relacionado). Usa en su lugar:

SH_ALLOW_DEPLOY=1 tools/wb-deploy.sh <script-sin-.js> "<mensaje>" [--minor|--set X.Y.Z]

wb-deploy.sh resguarda automáticamente la WIP de main (patch + git stash), aplica solo tu script, bumpea desde la versión ya commiteada de main, y restaura la WIP al final (con trap de recuperación si algo falla). SH_ALLOW_DEPLOY=1 es obligatorio: hay un guard de usuario (fuera de este repo, en ~/.claude/sh-workbench-guard.sh) que bloquea push/checkout de main desde un worktree que no sea el suyo, salvo que lo pases explícito. Es un script por corrida — si cambiaste varios, corre wb-deploy.sh una vez por cada uno.

08Deploy a producción

La extensión fetchea config.json + scripts/*.js desde GitHub Pages (rama gh-pages) en tiempo de ejecución. Desplegar = poner ahí el contenido nuevo, en sync byte a byte con main:remote/.

tools/deploy.sh "fix(applet-x): descripción"          # bump patch
tools/deploy.sh "feat(applet-x): ..." --minor          # bump minor
tools/deploy.sh "chore: ..." --set 1.7.0               # versión exacta
tools/deploy.sh "..." --check proceso-calculator       # verifica ese script tras publicar

Qué hace, en orden (leído directo del script): (1) localiza el worktree en main y exige que esté limpio salvo cambios bajo remote/, y al día con origin/main; (2) corre tools/run-tests.sh como gate — aborta si sale roja; (3) bumpea version/lastUpdated de remote/config.json; (4) sella scriptIntegrity + firma con KMS si SA_KMS_KEY está en el entorno (sección 9); (5) commit en main; (6) espeja main:remote/ → gh-pages (self-healing: re-sincroniza aunque hubiera drift previo) y commitea ahí; (7) push de ambas ramas; (8) crea un tag vX.Y.Z (ancla de rollback); (9) si pasaste --check, corre check-deploy.sh; (10) hace un smoke-check de la firma en vivo contra la pública embebida — si no verifica, el script sale con error antes de que te vayas confiado.

Solo se publican los scripts referenciados en config.apps[].scripts — los .js de remote/scripts/ que nadie referencia (helpers de build, módulos que solo usa un test) no se empujan a gh-pages.

Verificar el estado, sin adivinar

tools/deploy-status.sh

Imprime la versión de config.json en tu rama actual, en main, en gh-pages y en el sitio EN VIVO, y valida el invariante byte a byte. Es de solo lectura. Nunca concluyas el estado de deploy mirando el config.json de una rama de trabajo — puede estar desfasada respecto a main/gh-pages.

tools/check-deploy.sh [<script-name>]   # default: bulk-upload

Compara MD5 + version/VERSION local vs lo que GitHub Pages sirve ahora mismo, y te da un snippet para pegar en DevTools que confirma que Chrome ya cargó la versión nueva.

El candado: pre-push

Instalado por tools/install-hooks.sh (sección 4). Al pushear refs/heads/gh-pages, aborta si gh-pages no espeja main:remote/ byte a byte, si config.json version no sube respecto a lo publicado (excepto en pushes solo-docs, que no tocan config.json/scripts/), o si config.sig no verifica. No hay forma de saltártelo intencionalmente desde deploy.sh/wb-deploy.sh — si algo falla ahí, es una señal real de drift o de firma faltante, no un falso positivo a ignorar.

Rollback

tools/rollback.sh --list          # tags de deploy disponibles, más reciente primero
tools/rollback.sh v1.7.118        # revierte gh-pages al estado de ese deploy

Cada deploy exitoso queda anclado a un tag vX.Y.Z sobre el commit de main del bump — son, de facto, el CHANGELOG del proyecto (git tag -l 'v*' --sort=-version:refname). rollback.sh pide que escribas el tag para confirmar, y re-espeja <tag>:remote/ → gh-pages con el mismo mecanismo self-healing del deploy normal. Importante: solo revierte gh-pages — main sigue en el commit "malo", así que el siguiente deploy.sh volvería a introducir el problema si no arreglas main primero (git revert). Detalle completo en docs/architecture/rollback.md (Markdown, para consumo técnico).

09Firma y KMS (para poder desplegar)

Esto ya NO es opcional — verificado en el repo actual

La documentación de setup (docs/deploy-signing-setup.md) describe la firma como algo que se activa "cuando SA_KMS_KEY está seteada" y avisa que sin ella el deploy sale "sin firmar". Pero verifiqué el estado real de este repo hoy: la llave pública real (no un placeholder) ya está embebida en extension/integrity-pubkey.js, y remote/config.sig ya trae una firma real en gh-pages. Es decir, la Fase 2 (fail-closed) ya está viva. Consecuencia práctica: el chequeo de firma del hook pre-push (paso 4 del hook, sección 8) corre siempre y compara config.sig contra el config.json que estás pusheando. Si desplegás un cambio a remote/scripts/ o remote/config.json sin SA_KMS_KEY en el entorno, deploy.sh/wb-deploy.sh avisan "deploy SIN firmar" y siguen adelante — pero config.sig se queda desactualizado respecto al config.json nuevo, y el push origin gh-pages fallará bloqueado por el hook. En la práctica: no puedes publicar un cambio de código sin acceso a la llave KMS. (Los deploys solo-docs, como los del propio paquete de capacitación, no tocan config.json/scripts/ y por eso no disparan este problema.)

La llave privada vive en GCP KMS del proyecto del cliente (nunca en el repo ni en un laptop): proyecto steelhead-ecoplating, key ring steelhead-automator, llave config-signing v1 (EC_SIGN_P256_SHA256). El runbook completo de setup está en docs/deploy-signing-setup.md; aquí solo el resumen operativo:

Para que TÚ puedas firmar (pídeselo a quien tenga acceso hoy)

Quién puede darte de alta — ⚠️ confirmar con el equipo

Al corte de esta guía, el permiso roles/cloudkms.signerVerifier sobre la llave está concedido únicamente a oviazcan@gmail.com (confirmado en el comentario del .plist que agenda el hash-autopilot). Si eres nuevo en el equipo, necesitas que esa persona corra esto por ti — no puedes auto-otorgarte el permiso:

gcloud kms keys add-iam-policy-binding config-signing \
  --location=global --keyring=steelhead-automator --project=steelhead-ecoplating \
  --member="user:<tu-correo>" --role="roles/cloudkms.signerVerifier"

Deploy firmado, día a día

export SA_KMS_KEY="projects/steelhead-ecoplating/locations/global/keyRings/steelhead-automator/cryptoKeys/config-signing/cryptoKeyVersions/1"
tools/deploy.sh "fix(applet-x): descripción"   # seal + smoke-check automáticos

Requiere gcloud auth login (o application-default) con la cuenta que tiene el permiso de arriba. tools/seal-config.mjs calcula scriptIntegrity por script, firma con KMS, y produce config.sig — esto lo llama deploy.sh/wb-deploy.sh automáticamente, no lo corres a mano.

Al salir alguien del equipo (handoff)

gcloud kms keys remove-iam-policy-binding config-signing \
  --location=global --keyring=steelhead-automator --project=steelhead-ecoplating \
  --member="user:<correo-que-sale>" --role="roles/cloudkms.signerVerifier"

Eso es todo — sin re-generar la llave, sin transferir archivos, sin republicar la extensión.

10Hash-Autopilot (para mantener los hashes)

Detalle conceptual completo — con analogías y diagramas — en la Ficha profunda: Hash-Autopilot. Aquí, lo operativo para quien vaya a mantenerlo o correrlo a mano.

Es un proyecto Node/ESM independiente en tools/hash-autopilot/, con su propio package.json (única dependencia: playwright ^1.57.0). Abre un Chromium headless ya autenticado, navega/ejecuta las operaciones de Steelhead, captura los hashes de persisted queries que usa el frontend, y si detecta que uno rotó, lo valida y lo despliega solo.

cd tools/hash-autopilot
npm install
npm run install-browser     # playwright install chromium (cross-platform)

npm run dry-run              # clasifica, NO deploya ni notifica
npm start                    # auto-deploya + notifica (= node hash-autopilot.mjs)

# flags:
node hash-autopilot.mjs --dry-run --domain=344 --only=<Op> --date=YYYY-MM-DD --masked-only

Auth: depende del repo hermano "Reportes SH"

El motor inyecta tokens OAuth desde Reportes SH/.cache/tokens.json (mantenidos frescos por el flujo de steelhead_auth de ese otro repo — ver la skill steelhead-auth) en el localStorage del Chromium headless antes de cargar la app, para arrancar ya logueado sin flujo OAuth interactivo. Si el cache venció, el motor avisa "corre steelhead_auth.py" — ese script vive en el otro repo, no aquí.

Flag / variableEfecto
--dry-runClasifica todo pero no despliega ni notifica.
--masked-onlySolo re-captura las operaciones "sensibles a la sesión" (dan falso-stale al validador Python), sin depender de validate-hashes.py.
--only=<Op>Aísla el ciclo a una sola operación, para depurar.
SA_NO_EXTERNAL=1Desactiva la sincronización a los repos externos (Reportes SH / PowerTools).
SA_NO_EXTERNAL_PUSH=1Escribe y confirma en los repos externos, pero no hace push.

Agendado (macOS únicamente — launchd)

cp tools/launchd/com.ecoplating.steelhead-hash-autopilot.plist ~/Library/LaunchAgents/
launchctl load ~/Library/LaunchAgents/com.ecoplating.steelhead-hash-autopilot.plist
launchctl list | grep hash-autopilot

Corre cada hora (minuto :23). Un segundo .plist (com.ecoplating.steelhead-escalation.plist, minuto :53) dispara una escalación asistida por Claude cuando una ruta de navegación deja de capturar. launchd es exclusivo de macOS — no hay equivalente en el repo para agendar esto en Windows (Task Scheduler) o Linux (cron/systemd); si el plan es correr el autopilot agendado en otra plataforma, hay que construir ese agendador desde cero.

Un vigía separado corre en la nube de GitHub (no en la Mac), a propósito, para poder detectar si la Mac se apagó: .github/workflows/heartbeat-watchdog.yml revisa un latido que el motor empuja a la rama ops/heartbeat en cada corrida; si no lo ve en >3h, abre un issue.

11Diferencias Windows vs Mac

Todo lo que verifiqué que se comporta distinto entre los dos sistemas, en un solo lugar.

Las herramientas de tools/*.sh asumen macOS/BSD en dos puntos concretos

Verificado leyendo el código — no probado en vivo en Windows/WSL

tools/deploy.sh y tools/wb-deploy.sh bumpean la versión con sed -i '' -E "…" — sintaxis de edición-en-sitio de BSD sed (el propio comentario del script dice "sed in-place portable (macOS/BSD)"). GNU sed (el que trae Linux y WSL2 por default) interpreta -i '' distinto: el '' se consume como el patrón del script en vez de como sufijo de backup, así que el bump de versión no haría lo esperado corrido tal cual bajo GNU sed.

tools/check-deploy.sh usa md5 -q — el binario md5 con ese flag es de macOS; Linux trae md5sum en su lugar, con salida distinta.

El resto de los scripts que reviso esta guía (install-hooks.sh, new-worktree.sh, run-tests.sh, deploy-status.sh, rollback.sh) no usan estas llamadas específicas de BSD y deberían correr igual bajo Bash en Linux/WSL — pero no lo verifiqué ejecutándolos ahí. ⚠️ confirmar con el equipo antes de asumir que deploy.sh/wb-deploy.sh/check-deploy.sh funcionan sin ajuste desde WSL2; en la práctica hoy, el deploy real se hace desde macOS.

Macros VBA de la plantilla de carga masiva

La plantilla .xlsm (remote/templates/Plantilla_CargaMasiva_v12.xlsm y su variante _compatibilidad para Excel 2019) tiene diferencias de comportamiento documentadas en docs/applets/bulk-upload.md:

  • Sandbox de archivos de Excel para Mac. Al correr la macro "Refrescar Listas", Excel-Mac pide "conceder acceso" archivo por archivo cuando el VBA lee metadata (FileDateTime) de un archivo no autorizado — aunque no sea el que se va a abrir. Windows no sandboxea FileDateTime, así que ahí no pasa. Fix aplicado (Module2 v14, vbas/Module2.txt): elegir el catálogo más reciente por el nombre del archivo (fecha ISO en el nombre, ordenable como texto) en vez de por fecha de modificación en disco — así solo se toca el archivo ganador.
  • GrantAccessToMultipleFiles se evita a propósito: según el código, no habilita Open en esta configuración de Excel para Mac.
  • Pérdida de acentos en Mac. El VBA normaliza Ó→O porque Excel para Mac a veces pierde acentos en comparaciones de texto que Windows sí preserva.
  • BOM del CSV exportado. SaveAs FileFormat:=62 (CSV UTF-8) da terminadores de línea CRLF estables en ambos sistemas, pero Excel-Windows agrega un BOM UTF-8 (EF BB BF) al inicio del archivo y Excel-Mac a veces no. Si siempre exportas desde la misma máquina es consistente; si comparas/diffeas CSVs generados en máquinas distintas, los bytes iniciales pueden no coincidir.
  • Editar el VBA. El editor se abre con Alt+F11 en Windows; en Mac es Option (⌥)+F11 o el menú Herramientas → Macros. La fuente en texto plano vive en vbas/*.txt para diff/versión — el operador instala pegando el contenido en el módulo correspondiente del .xlsm.
  • Trasplante de vbaProject.bin. Como un .xlsm es un zip, un cambio que toca un solo módulo se puede propagar extrayendo/reinyectando el binario xl/vbaProject.bin entre copias del archivo (verificado antes con olevba -c que solo cambió el módulo esperado). Esto es una técnica ad hoc documentada una vez en la bitácora, no un script mantenido en tools/ — y no es transportable entre la plantilla normal y la de compatibilidad porque sus vbaProject.bin difieren en más de un módulo.
  • Los .xlsm de templates/ se despliegan a gh-pages a mano — deploy.sh y el hook pre-push solo gestionan config.json + scripts/**.js. Si tocas la plantilla, el commit en gh-pages debe ir en el mismo push que un bump de config.version (el hook exige que la versión suba en cada push a esa rama).

Shell y rutas

  • Todos los scripts de tools/ son Bash (#!/usr/bin/env bash). En Windows, córrelos desde Git Bash (incluido con Git for Windows) o WSL2 — no desde PowerShell ni cmd.exe directamente.
  • El nombre de carpeta del repo hermano Reportes SH trae un espacio — entrecomíllalo en cualquier comando de shell (cd "Reportes SH").
  • macOS (el shell de referencia de este repo, según CLAUDE.local.md) usa zsh por default desde Catalina; los scripts de tools/ declaran #!/usr/bin/env bash explícito así que corren igual con zsh como shell interactivo.

12Checklist de "entorno listo"

Antes de decir que tu entorno está montado

  • Clonaste el repo (git clone https://github.com/oviazcan/SteelheadAutomator.git) y — si vas a mantener el hash-autopilot — también Reportes SH y SteelheadPowerTools como carpetas hermanas.
  • Corriste tools/install-hooks.sh y confirmaste que pre-push existe en el directorio de hooks común del repo.
  • Node y Python 3 instalados; corriste node --version / python3 --version y los anotaste (no hay versión mínima fijada en el repo — documenta la tuya).
  • pip install pytest openpyxl hecho.
  • tools/run-tests.sh corre y sale verde, con la sección ▶ Python (pytest) ejecutando casos de verdad (no solo el aviso de "omito Python").
  • Cargaste la extensión descomprimida en Chrome (chrome://extensions → Modo desarrollador → Cargar descomprimida → extension/) y el popup dice "Conectado".
  • Corriste tools/deploy-status.sh al menos una vez para saber leer el ground-truth de versión antes de asumir que algo "ya está vivo".
  • Solo si vas a desplegar código: gcloud instalado, autenticado, con permiso roles/cloudkms.signerVerifier confirmado por quien administra hoy la llave; SA_KMS_KEY exportado en tu shell.
  • Solo si vas a mantener el hash-autopilot: tools/hash-autopilot/ con npm install + npm run install-browser hechos; confirmaste (o ajustaste) las rutas absolutas hardcodeadas a los repos hermanos.
  • Solo si vas a tocar la plantilla de carga masiva: Excel instalado, revisaste vbas/*.txt y entiendes las diferencias de sandbox Mac/Windows de la sección 11.
  • Leíste CLAUDE.md (raíz) completo al menos una vez — es el índice maestro y las reglas globales de desarrollo.
