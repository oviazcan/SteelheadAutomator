AArquitectura y despliegue

Cómo está armado el sistema por dentro y cómo un cambio llega de la computadora del equipo a la extensión que usan los operadores.

Applet

SteelheadAutomator

Cada herramienta individual dentro de la extensión —Carga Masiva, Auto-Ruteador, Candado de Confirmación de Precio, etc. Es un archivo (o un grupo pequeño de archivos) registrado en config.json, con su propio número de versión y su propia bitácora.

Cache-bust / versión

Infraestructura

Técnica para forzar a que el navegador descargue la versión nueva de un archivo en vez de servir una copia vieja que tenía guardada. En SteelheadAutomator es el campo version de config.json: cada vez que sube, la extensión sabe que debe recargar los scripts en vez de reusar los que ya tenía en memoria.

Cascarón / remote script loader

SteelheadAutomator

La extensión instalada en Chrome casi no trae lógica propia: al arrancar, descarga los scripts reales desde GitHub Pages y los ejecuta. Eso permite corregir o mejorar un applet sin pedirle a nadie que reinstale nada.

AnalogíaUn decodificador de TV que baja su firmware del satélite cada vez que se enciende: la caja física casi no cambia, pero la inteligencia —qué canales hay, cómo se ven— vive del lado del servidor, no en el aparato.

config.json

SteelheadAutomator

El catálogo central del sistema: qué applets existen, qué archivos de código carga cada uno, los hashes vigentes de cada operación contra el ERP, y el número de versión que actúa como cache-bust. Vive publicado en gh-pages.

Deploy / despliegue

SteelheadAutomator

El proceso que toma lo que ya se probó en main y lo publica en gh-pages: sube el número de versión, mueve el código, y sube ambas ramas. Se corre con un script dedicado (tools/deploy.sh), nunca a mano, y deja un tag por cada versión para poder revertir si algo sale mal.

Documento vivo vs. perenne

SteelheadAutomator

Vivo: refleja el estado actual del sistema y se espera que cambie seguido (versiones, conteos, inventarios). Perenne: documenta un concepto o un mecanismo que no cambia con cada deploy —como este glosario—, aunque las cifras de ejemplo que trae sí puedan quedar viejas con el tiempo.

Extensión de Chrome (MV3)

Infraestructura

El formato de extensión que usa Chrome hoy en día (Manifest Version 3). Define reglas de seguridad y de ciclo de vida que SteelheadAutomator respeta —y que es la razón por la que su lógica principal vive fuera del paquete instalado (ver Cascarón).

gh-pages / rama publicada

SteelheadAutomator

La rama del repositorio cuyo contenido sí llega a los operadores: es lo que la extensión descarga en tiempo real desde GitHub Pages. Editar código en otra rama no afecta a nadie hasta que se despliega aquí.

Analogíamain es la cocina de pruebas —ahí se experimenta, se corrige, se cocina de nuevo si algo sale mal— y gh-pages es el menú que sí se sirve en la mesa. Un platillo en la cocina no le llega a nadie hasta que pasa al menú.

GitHub Pages

Infraestructura

El servicio gratuito de GitHub que sirve archivos estáticos (HTML, JS, JSON) por HTTPS a partir del contenido de una rama del repositorio. Es el "servidor" del que la extensión descarga sus scripts y su config.json.

Golden test

Infraestructura

Una prueba automatizada que compara el resultado de una función contra un resultado ya conocido y aprobado de antemano. Si algo cambia sin querer, la prueba se pone roja antes de que ese cambio llegue a producción. Cada núcleo puro del sistema trae su propio set de golden tests.

AnalogíaEs una foto de referencia: si la foto nueva no se parece a la de referencia, algo cambió —y hay que revisar si fue a propósito o fue un error.

main / rama de desarrollo

SteelheadAutomator

La rama donde se escribe y se prueba el código antes de publicarlo. Editar aquí no afecta a ningún operador —ese es justo el punto— hasta que un deploy lo lleva a gh-pages.

Núcleo puro (core) / módulo puro

SteelheadAutomator

La parte de un applet que hace el cálculo o toma la decisión, sin tocar la pantalla ni la red: mismos datos de entrada, mismo resultado siempre. Al no depender de Steelhead ni del navegador, se puede probar con un golden test en segundos, sin abrir el ERP.

Orquestador

SteelheadAutomator

La otra mitad de un applet: conecta el núcleo puro con el ERP (llamadas GraphQL) y con la pantalla (paneles, botones, lectura del DOM de Steelhead). El núcleo decide "qué hacer"; el orquestador se encarga de "hacerlo" de verdad.

BConexión con el ERP

Cómo le habla la extensión a Steelhead, y qué pasa cuando esa conversación se corta.

Apollo client

Infraestructura

La librería que el propio frontend de Steelhead usa para hablar con su API de GraphQL. La extensión debe identificarse con la misma versión de Apollo client en sus peticiones para que el servidor la trate como tráfico normal de la aplicación.

Captura-y-aborta

SteelheadAutomator

Técnica para investigar una operación de escritura delicada sin arriesgar datos reales: se deja que el navegador arme la petición completa —así se ve el código que va a usar— pero se intercepta y se cancela antes de que llegue al servidor. Nada queda guardado.

AnalogíaEs como pasar por la caja registradora un producto marcado "PRUEBA": se escanea, se confirma que el sistema lo reconoce, y se anula el ticket antes de que se cobre.

Cookie de sesión

Infraestructura

El mecanismo de autenticación real contra Steelhead. La extensión no tiene usuario ni contraseña propios: usa la misma sesión que ya abriste en tu navegador para entrar a Steelhead. Si esa sesión expira, la extensión también deja de poder consultar datos.

ERP / Steelhead

Steelhead

El sistema (app.gosteelhead.com) donde vive toda la operación real: clientes, órdenes, inventario, procesos, facturación. SteelheadAutomator no lo reemplaza ni tiene su propia base de datos de producción: automatiza tareas dentro de él, usando su misma API.

GraphQL

Infraestructura

El lenguaje de consulta que usa la API de Steelhead. En vez de tener una dirección web distinta por cada tipo de dato, todo pasa por un único punto de entrada (/graphql) donde se especifica exactamente qué información se necesita.

Hash / persisted query

Steelhead

Un código corto (64 caracteres) que identifica una consulta completa ya registrada en el servidor de Steelhead. En vez de mandar la pregunta entera cada vez, la extensión manda solo el código; el servidor sabe qué pregunta completa corresponde a ese código.

AnalogíaCada operación del ERP es una puerta. El hash es la llave que la abre: no hace falta describir la puerta cada vez, basta con presentar la llave correcta.

Hash-Autopilot

SteelheadAutomator

El proceso automatizado que detecta cuándo una llave dejó de abrir su puerta, deduce la llave nueva navegando Steelhead como lo haría un usuario real, la valida, y la publica sola en la extensión (y en los otros proyectos que comparten hashes) —sin que nadie tenga que capturarla a mano.

AnalogíaEs el cerrajero automático: no espera a que alguien reporte la cerradura cambiada, prueba las llaves, nota cuál dejó de servir y saca una copia nueva por su cuenta.

"Must provide a query string." (HTTP 400)

Steelhead

El error exacto que Steelhead devuelve cuando una llave (hash) ya no abre ninguna puerta. Es la señal de que hubo una rotación —o, más raro, de que esa operación se retiró por completo y no hay llave nueva que capturar.

Rotación de hash

Steelhead

Cuando Steelhead actualiza su software y una consulta que antes tenía un código pasa a tener uno distinto. El código viejo deja de funcionar de un momento a otro, sin aviso previo. No es un error nuestro: es cómo funciona la técnica de persisted queries del lado del ERP.

AnalogíaSteelhead cambió la cerradura. La llave vieja es la misma de siempre; simplemente ya no abre esa puerta.

Centinela

SteelheadAutomator

Un objeto real dentro de Steelhead —una pieza, una cotización, una orden— marcado literalmente con el nombre "Centinela" y dedicado a que el hash-autopilot dispare operaciones de escritura sin tocar datos de producción. Antes de usarlo, el sistema verifica que su nombre siga siendo exactamente ese; si no, se detiene solo.

Validador de hashes

SteelheadAutomator

El script que prueba, uno por uno, cada llave conocida contra el servidor de Steelhead y anota cuáles ya no responden. Es la primera etapa del hash-autopilot: detecta el corte antes de intentar repararlo.

CSeguridad

Cómo se protege el código que la extensión descarga y ejecuta cada vez que arranca.

Break-glass

SteelheadAutomator

Un interruptor de emergencia, apagado por default, para desactivar la verificación de firma si un despliegue legítimo se queda bloqueado por error. Vive en la configuración de la extensión y solo lo activa la extensión misma; el arreglo real de fondo siempre es volver a desplegar con una firma correcta.

Fail-closed

SteelheadAutomator

Principio de diseño: ante cualquier duda —una firma que no valida, un dato que falta, una identidad que no se puede confirmar— el sistema no ejecuta nada, en vez de arriesgarse a correr algo indebido. Es el criterio detrás de la verificación de scripts y de las salvaguardas del hash-autopilot.

Firma criptográfica / ECDSA P-256

SteelheadAutomator

Cada script que la extensión descarga de GitHub Pages viene firmado con una llave privada; la extensión trae la llave pública y, antes de ejecutar cualquier código, verifica que la firma coincida. Si alguien alterara el contenido publicado sin volver a firmarlo, la extensión se niega a correrlo.

AnalogíaEs un sello de lacre sobre el sobre: si el sello no coincide o está roto, no se abre.

Firma vigente desde v1.6.6

SteelheadAutomator

La extensión empezó a verificar la firma de cada script antes de ejecutarlo a partir de la versión 1.6.6 (julio 2026). Es una protección gradual: solo queda activa fail-closed en las computadoras que ya actualizaron a esa versión o una posterior.

KMS (bóveda de llaves)

Infraestructura

Key Management Service: el servicio de Google Cloud donde vive la llave privada que firma los scripts. Nunca sale de ahí ni se copia a ningún equipo local —firmar es pedirle al servicio "fírmame esto", no tener la llave en la mano.

AnalogíaLa llave privada vive en la bóveda de Google: nadie la saca de ahí, solo se le pide que la use.

DCarga Masiva

El applet más complejo del sistema tiene su propio vocabulario, sobre todo alrededor de cómo decide qué tan a fondo procesar cada corrida.

Batching / lotes de 20

SteelheadAutomator

En vez de mandar una petición al ERP por cada número de parte, Carga Masiva agrupa hasta 20 por llamada. Menos viajes de red, menos probabilidad de saturar al servidor y de terminar en un rate-limit.

classifyRunIntent / intención de corrida

SteelheadAutomator

Antes de procesar nada, Carga Masiva mira el archivo completo y decide qué tipo de corrida es: solo cambia precios, solo da de alta números de parte nuevos, o trae enriquecimiento real (specs, dimensiones, etiquetas). Esa decisión determina qué tan a fondo se procesa cada fila.

AnalogíaEs triage médico: antes de tratar, se clasifica qué tan grave es el caso, para no darle el mismo tratamiento completo a todos.

Fast-path SOLO_PRECIO

SteelheadAutomator

Cuando classifyRunIntent determina que una corrida solo cambia precios de números de parte que ya existen —cero enriquecimiento—, el atajo salta por completo el paso de enriquecimiento (que si no, revisaría cada pieza una por una) e va directo a guardar los precios. Trae dos invariantes de seguridad: una sola fila con enriquecimiento, o un solo número de parte nuevo, desactivan el atajo.

AnalogíaEs el carril express de aduana: si de verdad no traes nada que declarar, no pasas por la revisión completa.

HTTP 429 / rate-limit

Steelhead

El código que devuelve el servidor de Steelhead cuando se le pide demasiado, demasiado rápido: "más despacio". Corridas muy grandes, o que no agrupan sus peticiones en lotes, son las que más lo disparan.

IndexedDB (Historial de Cargas)

Infraestructura

La base de datos que vive dentro del propio navegador. Carga Masiva la usa para guardar el historial de corridas —qué se cargó, cuándo, con qué resultado— sin depender de un servidor propio del equipo.

REEMPLAZO (semántica de la API)

Steelhead

Cuando una mutación de Steelhead (como SavePartNumber) recibe un arreglo, no lo combina con lo que ya existía: lo sustituye completo. Mandar una lista vacía no significa "no cambies nada" — significa "borra todo lo que había ahí". Ignorar esta regla es la causa raíz detrás de varios bugs corregidos en Carga Masiva (por ejemplo, opt-ins de validación borrados sin querer).

Retry / backoff / jitter

Infraestructura

La estrategia para reintentar una petición que falló sin empeorar el problema: esperar un poco más cada vez que vuelve a fallar (backoff) y variar ligeramente esa espera entre peticiones (jitter) para que no todas reintenten en el mismo instante exacto y saturen otra vez al servidor.

EOperación y datos

Los objetos de negocio de Steelhead sobre los que trabajan casi todos los applets.

Cotización

Steelhead

Una propuesta de precio a un cliente, antes de convertirse en una orden confirmada. En Steelhead comparte pantalla y flujo con la creación de órdenes de venta —de ahí que Carga Masiva y otros applets hablen indistintamente de "cotizaciones y números de parte".

DuckDB (reportes)

Infraestructura

El motor de base de datos analítica donde se regenera periódicamente un snapshot de los datos de Steelhead, para poder correr reportes pesados sin sobrecargar el ERP en vivo. El applet Regenerar Reportes dispara esa regeneración desde la propia pantalla de Steelhead.

Número de parte (PN)

Steelhead

La unidad central de Steelhead: una pieza o artículo específico de un cliente, con sus precios, especificaciones, parámetros y proceso de producción asociado. La mayoría de los applets del sistema leen o escriben directamente sobre números de parte.

Orden de trabajo (OT/WO)

Steelhead

La instrucción de producción real: qué pieza, con qué proceso, en qué línea. Las órdenes de trabajo se agrupan bajo una orden de venta y se mueven entre líneas y estaciones conforme avanza la producción —de ahí applets como el Auto-Ruteador o Mover Órdenes de Trabajo.
