Administración y operación del sistema

Manual del Administrador

Arquitectura, deploy, rotación de hashes, firma criptográfica, distribución y respuesta a incidentes — lo que el Jefe de TI necesita para operar y responder por SteelheadAutomator sin depender de quien lo construyó.

Jefe de TI Documento perenne

En una frase

SteelheadAutomator es un cascarón instalado en Chrome que en cada arranque baja su “cerebro” —los scripts que hacen el trabajo— desde GitHub Pages, verifica su firma criptográfica antes de ejecutarlo, y habla con el ERP de Steelhead con las mismas credenciales de sesión del operador. Publicar un cambio es un deploy.sh; revertirlo es un rollback.sh; y la avería más común del sistema —un código de operación que Steelhead “rota”— se repara sola.

01Panorama para el administrador

SteelheadAutomator automatiza tareas repetitivas dentro de Steelhead ERP (app.gosteelhead.com) — carga masiva de cotizaciones, números de parte, ruteo, validaciones de captura, y otras ~41 herramientas (“applets”). No es una aplicación aparte con su propio backend: es una extensión de Chrome que inyecta código dentro de la pestaña donde el operador ya tiene sesión abierta, y ese código habla con el mismo servidor GraphQL que usa la interfaz nativa de Steelhead.

Para el Jefe de TI, esto tiene tres implicaciones directas:

  • No hay backend propio que mantener. No hay servidor, base de datos ni sesión que administrar aparte de la del navegador — la autenticación es la cookie de sesión que el operador ya tiene al usar Steelhead.
  • El código que corre en la máquina del operador no vive fijo en su máquina. Se descarga en cada arranque desde un hosting externo (GitHub Pages). Eso hace los arreglos instantáneos (no hay que reinstalar nada), pero también significa que la disponibilidad de ese hosting es una dependencia real del sistema (ver §8).
  • La superficie de ataque relevante es la integridad del código remoto, no una base de datos o un login propios. Por eso el control central de este manual es la firma criptográfica (§6).

Fuera de alcance de este documento

Este manual cubre administración y operación del sistema ya construido: deploy, rollback, hashes, firma, distribución, incidentes. El montaje del entorno de desarrollo (clonar el repo, instalar dependencias, correr la suite de pruebas local, configurar el cron de Hash-Autopilot desde cero) se cubre en la Guía de Onboarding aparte, dirigida a quien va a desarrollar o dar mantenimiento al código.

02Arquitectura: cascarón + cerebro remoto

La pieza conceptual que explica casi todo lo demás en este manual.

Analogía — el decodificador de TV

La extensión instalada en Chrome es como un decodificador de TV de paga: la caja física en casa del operador no “sabe” programas, solo sabe bajar el firmware y ejecutarlo. La inteligencia —qué botones muestra, qué automatiza cada applet, qué validaciones aplica— vive en un servidor remoto (aquí, GitHub Pages). Actualizar el decodificador no requiere ir a la casa del operador: se actualiza el firmware del lado del proveedor y la caja lo recoge en el siguiente encendido.

Las dos ramas que importan

Todo el repositorio vive en git, pero para el administrador solo dos ramas tienen efecto operativo:

RamaQué es¿Afecta a usuarios?
mainRama de desarrollo. Aquí se edita remote/ (scripts + config.json).No. Es “cocina de pruebas” — nada de lo que se cambia aquí llega a un operador hasta el siguiente paso.
gh-pagesRama publicada. GitHub Pages sirve su contenido en https://oviazcan.github.io/SteelheadAutomator/.Sí, en vivo. Es “el menú que sí se sirve”: lo que está aquí es literalmente lo que la extensión de cada operador va a descargar y ejecutar en su próximo arranque.

gh-pages debe quedar en sincronía byte a byte con el contenido de remote/ de main en el momento del deploy — no es una copia aproximada, es un espejo exacto. Ese invariante es lo que verifica tools/deploy-status.sh (§3).

03Del borrador a producción

Publicar un cambio es un solo comando atómico. Nunca se edita gh-pages a mano.

tools/deploy.sh "fix(applet-x): descripción del cambio"

Ese único comando hace, en orden, toda la secuencia:

  1. 1

    Gate de pruebas

    Corre la suite de tests del repo (tools/run-tests.sh). Si algo falla, el deploy se aborta ahí mismo — nada llega a producción con pruebas rojas.

  2. 2

    Cache-bust

    Sube el campo version de remote/config.json (p. ej. 1.7.176 → 1.7.177). Ese número es la señal que hace que cada extensión instalada note que hay algo nuevo que descargar y recargue sus scripts — sin ese bump, el navegador podría seguir sirviendo la copia en caché.

  3. 3

    Firma con KMS

    Calcula la huella SHA-256 de cada script y firma config.json con la llave privada en Google Cloud KMS (produce config.sig). Detalle completo en §6.

  4. 4

    Commit en main

    Deja registrado el cambio de código con el mensaje que se le dio al comando.

  5. 5

    Espejo a gh-pages

    Copia main:remote/ → la raíz de gh-pages byte a byte (estructura aplanada: remote/scripts/foo.js → scripts/foo.js). Es self-healing: si gh-pages tenía drift de un deploy manual anterior, este paso lo corrige también, no solo aplica el diff del cambio actual.

  6. 6

    Push de ambas ramas + tag

    Sube main y gh-pages, y crea un tag vX.Y.Z sobre el commit de main — ese tag es el ancla de rollback (§4) y, acumulado, el changelog del sistema.

  7. 7

    Verificación en vivo

    Espera la propagación de GitHub Pages y confirma contra el sitio real que la versión nueva quedó publicada y que la firma verifica con la llave pública embebida.

Tiempos de propagación

GitHub Pages publica el contenido nuevo en ~30 a 60 segundos después del push. Cada extensión instalada lo recoge en su siguiente arranque de pestaña (o al recargarla manualmente desde chrome://extensions) gracias al cache-bust del paso 2. No hay que reinstalar nada del lado del operador.

Verificar el estado sin adivinar

Antes de asumir si un cambio “ya está vivo”, la pregunta se responde con un comando, no mirando el config.json de una rama de trabajo cualquiera (puede estar desfasado):

tools/deploy-status.sh

Imprime cuatro números de versión —el de la rama actual, el de main, el de gh-pages, y el que está realmente sirviendo GitHub Pages ahora mismo— y verifica que el invariante byte a byte entre main:remote/ y gh-pages se cumpla. Si algo no cuadra, lo señala explícitamente en vez de dejarlo pasar en silencio.

Regla dura

gh-pages nunca se edita a mano. Hay un hook pre-push instalado (.githooks/pre-push) que bloquea el push si gh-pages no espeja exactamente main:remote/, o si la firma no verifica. El único camino soportado es deploy.sh (o, para equipos de desarrollo aislados en un worktree, su variante wb-deploy.sh — detalle de desarrollo, fuera de alcance de este manual).

04Rollback

Cada bump de versión deja un ancla. Revertir es tan atómico como publicar.

Cada corrida de deploy.sh crea un tag de git vX.Y.Z sobre el commit exacto de main que se publicó. Ese tag contiene el remote/ completo tal como quedó en producción en ese momento — es, acumulado, el changelog del sistema: no hace falta mantener un archivo aparte, la lista de tags con sus mensajes de commit ya lo es.

tools/rollback.sh --list          # tags de deploy disponibles, más reciente primero
tools/rollback.sh v1.7.118        # revierte gh-pages al estado exacto de ese deploy

rollback.sh valida el tag, muestra la versión viva contra el destino y pide confirmar escribiendo el tag antes de tocar nada. Después re-espeja <tag>:remote/ → gh-pages con el mismo mecanismo self-healing de deploy.sh (elimina scripts huérfanos que no existían en ese tag) y hace commit + push solo de gh-pages.

El rollback no toca main

main sigue apuntando al commit que causó el problema. Si el siguiente deploy.sh corre sin corregir eso primero, re-introduce el bug (porque vuelve a espejar main:remote/). Antes de deployar de nuevo hay que revertir o corregir el commit malo en main. El rollback del .zip de la extensión empaquetada (si el deploy malo tocó extension/) es un procedimiento aparte — republicar el zip de la versión anterior — porque el .zip no se mueve con deploy.sh.

05Hashes y su rotación

La avería más frecuente del sistema, y la única que se repara sola.

Analogía — la llave y la cerradura

Para pedir datos al ERP, la extensión no manda la pregunta completa cada vez: manda un código corto (un hash SHA-256, técnica llamada persisted queries) que el servidor de Steelhead ya asocia a esa consulta exacta. Ese código es una llave que abre una puerta específica. Cuando Steelhead actualiza su propio software, a veces cambia la cerradura: el código viejo deja de servir y la respuesta es HTTP 400 "Must provide a query string." — en español, “no reconozco lo que me estás pidiendo”.

Esto no es un bug de SteelheadAutomator: es consecuencia de cómo Steelhead implementa su API. La reparación la hace un proceso automático propio: Hash-Autopilot, un “cerrajero automático” que corre cada hora y, en el caso común, sin intervención humana (aunque no cubre el 100% de los casos — ver §9):

  1. Detecta qué códigos ya no responden (un validador los prueba contra el servidor real).
  2. Re-captura el código nuevo abriendo un navegador oculto, ya logueado, y navegando la ruta mínima que dispara esa operación (o, para escrituras, usando objetos de prueba marcados “Centinela” con captura-y-aborta: arma la petición para ver su código, pero la cancela antes de que se guarde nada real).
  3. Valida que el código nuevo es real, no ruido.
  4. Publica el arreglo con commit y push automáticos — a la extensión, y también a los dos repos hermanos que comparten hashes (Reportes SH y PowerTools).
  5. Notifica por correo qué se corrigió, qué quedó pendiente de revisión humana, y qué es urgente.

Tiene frenos de seguridad: si en una sola corrida parecen romperse más de 6 códigos a la vez, no despliega nada y pide revisión humana (eso casi siempre significa una captura corrupta, no una rotación real). Y manda un “latido” a la nube en cada corrida — si un vigía externo no lo ve en más de 3 horas, avisa por correo de que el robot pudo haberse detenido.

Detalle completo

El mecanismo interno de Hash-Autopilot (las cinco etapas, la deduplicación entre los tres proyectos, los dos caminos lectura/escritura, todas las salvaguardas y las banderas de control) está documentado a fondo en la Ficha profunda de Hash-Autopilot. Este manual solo cubre lo que el Jefe de TI necesita saber para confiar en que el sistema se repara solo y reconocer cuándo no lo hace (§9).

06Seguridad: la firma criptográfica

La extensión ejecuta código descargado de internet en cada arranque. Esto es lo que impide que ese código sea confiable solo porque “vino del repo correcto”.

Analogía — el sello de lacre

Cada publicación de config.json lleva un sello de lacre: una firma ECDSA que solo se puede producir con la llave privada, guardada en la bóveda de Google (Cloud KMS, en el proyecto GCP del cliente, nunca en el repositorio). La extensión trae la llave pública embebida y, antes de ejecutar cualquier script, rompe el sello: si la firma no cuadra con lo que descargó, no ejecuta nada — ni siquiera actualiza su copia en caché. Ante la duda, la puerta se queda cerrada (fail-closed).

Los componentes

PiezaRol
Llave privadaVive en Google Cloud KMS, en el proyecto GCP del cliente (no de quien desarrolla). El handoff de esta pieza es un cambio de permisos IAM, no una transferencia de archivo.
Llave públicaEmbebida dentro de la extensión (extension/integrity-pubkey.js). Es la raíz de confianza del lado del cliente.
config.sigArchivo separado con la firma de config.json, publicado junto a él en gh-pages.
scriptIntegrityDentro de config.json: una huella SHA-256 por script. Una vez que la firma del config.json verifica, esas huellas quedan confiables y cada script se valida contra la suya antes de ejecutarse.
Break-glassInterruptor manual en el popup de la extensión (Configuración → Seguridad) para desactivar la verificación en un incidente. Apagado por default. Es la última línea de defensa — el arreglo real de un incidente de firma es volver a publicar con una firma correcta, no dejar el break-glass encendido.

Caveat de versión — la protección no es universal todavía

La verificación fail-closed solo está activa a partir de la extensión v1.6.6 (versión en la que se embebió la llave pública real y se republicó el paquete). Cada equipo queda protegido al actualizar a esa versión o una posterior — el rollout es gradual, máquina por máquina, disparado por el banner de actualización (§7). Los equipos que sigan en una versión anterior a v1.6.6 operan en modo fail-open: no verifican firma, igual que antes de que este control existiera. Confirmar que el parque de equipos está en v1.6.6+ es una tarea de administración recurrente, no un evento único.

07Distribución y actualización de la extensión

La extensión se distribuye como un archivo .zip de instalación manual (“unlisted”), publicado en gh-pages, con una guía de instalación (install-guide.html) para quien la instala por primera vez.

Por qué no está en la Chrome Web Store

La arquitectura de carga remota (new Function sobre código bajado en runtime) es incompatible con las políticas de revisión de la Chrome Web Store para MV3. Es una decisión de arquitectura consciente, no un pendiente — la distribución vía .zip es el canal soportado.

El campo extensionVersion dentro de config.json es el disparador del banner “actualiza tu extensión” que ve el operador. Ese campo solo cambia cuando cambió código dentro de extension/ (el cascarón mismo) y se republicó el .zip — no cada vez que se deploya un applet nuevo en remote/ (eso lo cubre el cache-bust de version, §3, y no requiere que el operador reinstale nada).

En otras palabras: hay dos ciclos de actualización distintos y es importante no confundirlos como administrador:

  • Cambios en remote/ (la mayoría): invisibles para el operador salvo por el efecto del applet — se recogen solos en el siguiente arranque de pestaña.
  • Cambios en extension/ (poco frecuentes): requieren que el operador vea el banner, descargue el .zip nuevo y lo recargue manualmente en chrome://extensions. Es el único camino por el que un equipo llega a v1.6.6+ y activa la protección de firma (§6).

08Dependencias y modos de falla

Tres sistemas externos de los que SteelheadAutomator depende para operar. Ninguno lo controla el equipo de desarrollo.

DependenciaPara qué se usaSi fallaCobertura
GitHub PagesHospeda el “cerebro” — config.json, config.sig y todos los scripts.La extensión no tiene qué descargar ni ejecutar; ningún applet funciona hasta que Pages vuelva a responder.Sin mitigación automática — es la razón por la que el repositorio no debe volverse privado sin antes migrar el hosting: eso rompería la extensión para todos.
API GraphQL de SteelheadEl ERP mismo — cada operación que hace un applet.La rotación de hashes (§5) es el modo de falla más común y ya está cubierto. Caídas totales del ERP no son responsabilidad de SteelheadAutomator.
Google Cloud KMSFirma cada deploy (§6).Sin acceso a KMS, deploy.sh aborta el paso de firma (o publica sin firmar, si así se configura explícitamente) — no bloquea deploys de emergencia, pero deja esa publicación sin el sello de confianza.Acceso IAM administrado por el cliente; el handoff de esta pieza es exactamente eso: permisos, no archivos.

Monitoreo activo

Dos procesos corren sin intervención y son la primera línea de defensa contra estos modos de falla:

  • Hash-Autopilot (cron horario) — detecta y repara rotaciones de hash; manda su propio “latido” a un vigía que avisa si deja de correr.
  • Validador diario de hashes — corre independiente del autopilot, prueba cada operación conocida contra el servidor real y notifica por correo si algo quedó roto.

09Playbook de incidentes

Síntoma: "Must provide a query string." (HTTP 400)

Un operador reporta que un applet “dejó de funcionar de repente” y, al revisar la consola o el error mostrado, aparece exactamente ese mensaje.

  1. Diagnóstico: esto es un hash rotado (§5), no un bug del código de SteelheadAutomator. Es esperado que ocurra de tanto en tanto — es consecuencia de cómo Steelhead versiona su API.
  2. Primera acción: no acción. Hash-Autopilot corre cada hora; en la mayoría de los casos, el problema se resuelve solo antes de que un humano intervenga.
  3. Confirmar que el sistema está reaccionando: correr tools/deploy-status.sh para ver la versión publicada más reciente. Si hay un deploy de Hash-Autopilot posterior a la hora del reporte, ya está en camino o resuelto.
  4. Si pasan varias horas sin resolverse, revisar el correo de notificación de Hash-Autopilot — puede haber activado el freno de masa (más de 6 rotaciones simultáneas) o encontrado una deprecación dura: Steelhead retiró la consulta por completo, no solo le cambió el código. Eso sí requiere intervención humana — migrar el applet a una consulta equivalente que siga viva.
  5. Verificar el latido: si el vigía de Hash-Autopilot no ha visto un “latido” reciente, el proceso mismo pudo haberse detenido (p. ej. la máquina que lo corre se apagó) — ese es un incidente de infraestructura distinto al de un hash rotado, y sí necesita atención directa.

Cuándo interviene un humano de verdad

  • Deprecación dura de una operación (no hay código nuevo que capturar porque la consulta ya no existe en el servidor).
  • Ruta de navegación rota: Hash-Autopilot re-captura hashes de lectura navegando pantallas reales; si Steelhead rediseña una pantalla, esa ruta puede dejar de disparar la operación esperada. El sistema intenta una escalación asistida antes de rendirse, pero puede requerir ajuste manual de la ruta.
  • Freno de masa activado (>6 rotaciones de golpe en una sola corrida) — casi siempre señal de una captura corrupta (sesión equivocada, por ejemplo), no de una rotación legítima masiva. Requiere revisión antes de permitir el siguiente deploy automático.
  • Firma que no verifica tras un deploy — el smoke-check de deploy.sh lo señala; no se debe activar el break-glass como solución permanente, sino re-publicar con la firma correcta.
  • GitHub Pages o KMS caídos — fuera del control del sistema; escalar al proveedor correspondiente.

Cómo recuperar un hash a mano

Cuando el robot no puede (cualquiera de los casos de arriba), el procedimiento concreto —recuperar el hash con el escáner de Steelhead y entregárselo a Claude para que lo ajuste y lo despliegue— está paso a paso en el Runbook — recuperar hashes a mano (escáner + Claude).

10Lo esencial para el Jefe de TI

Si solo te quedas con esto

  • No hay backend propio. La extensión usa la cookie de sesión del operador contra app.gosteelhead.com/graphql. La superficie a administrar es la integridad del código remoto, no una base de datos.
  • Publicar = tools/deploy.sh. Atómico: pruebas → cache-bust → firma KMS → commit → espejo a gh-pages → push → tag → verificación en vivo. Nunca se edita gh-pages a mano (hay un hook que lo bloquea).
  • Revertir = tools/rollback.sh <tag>. Cada deploy deja un tag; los tags son el changelog. El rollback solo toca gh-pages — main hay que corregirlo aparte.
  • Verificar estado = tools/deploy-status.sh. Nunca concluir el estado de un deploy mirando el config.json de una rama de trabajo.
  • El código está firmado con ECDSA P-256, llave privada en KMS del cliente, verificación fail-closed en la extensión — pero solo desde la v1.6.6. Confirmar que el parque de equipos está actualizado es tarea recurrente de administración, no un evento único.
  • El síntoma más común es HTTP 400 "Must provide a query string." — un hash rotado. Se repara solo (Hash-Autopilot, cron horario); casi nunca requiere intervención humana. Cuando sí la requiere, es porque Steelhead deprecó algo por completo o rediseñó una pantalla que rompió una ruta de captura.
  • La dependencia externa que sí puede tumbar todo es GitHub Pages: si el hosting del cerebro remoto desaparece, la extensión no tiene qué ejecutar. No volver privado el repositorio sin antes resolver esto.
