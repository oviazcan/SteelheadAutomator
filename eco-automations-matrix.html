Eco Plating — Browser Automations

Capability & Risk Matrix — prepared for Steelhead as input to the upcoming “Automations” feature.

This matrix lists every browser-level tool Eco Plating uses today on top of Steelhead. It is organized around the two dimensions we agreed to evaluate: data-safety footprint and operational importance. These tools were built as a value-add to accelerate our Steelhead adoption and to cover operational needs the platform does not yet handle natively. Our intent in sharing this is to give your team a clear road map so the native “Automations” feature can absorb as much of this as possible.

How these tools touch Steelhead (shared safety model)

The following holds for every tool below and is the basis for the per-tool data-safety rating:

  • Same session, same permissions. Everything runs inside the signed-in user’s own browser session, with that user’s existing role and permissions. The tools do only what the same person could do by hand in the UI — no admin/elevated access.
  • Platform’s own operations. They reuse the very same operations the Steelhead web app itself issues; they do not open any private or undocumented channel. Your server-side validation and safeguards stay fully in effect.
  • Conservative request volume. Bulk work is deliberately throttled (small batches, capped concurrency) to be gentle on the platform — not a flood of uncontrolled calls.
  • Mostly reversible & auditable writes. The large majority of writes are reversible (archive, not hard-delete) and carry an audit record (who / when / what). The few higher-impact tools are marked † below and are explicitly gated behind a user confirmation step.
  • Proactive break detection. An automated daily check detects when Steelhead rotates its operation identifiers, so affected tools are flagged and fixed quickly — usually before a user hits an error.

† Higher-impact operations (each requires explicit confirmation before running): inventory-reset (archives existing batches + creates new ones), process-canon apply mode (rewrites a process node tree via the platform’s own ProcureTree), and report-liberator’s optional empty-folder delete.

At a glance

  • 30 tools in the extension launcher + 5 standalone DevTools = 35 in use, across 7 categories — plus 4 high-priority capabilities we’ve identified but not yet built (engineering-calc, tank-autoroute, so-line-merge, warehouse-issue-slip).
  • The clear majority address gaps Steelhead has no native equivalent for today. Where it’s marked “Partial,” the function exists but is per-record, buried in the UI, or has no bulk path.
  • Data-safety footprint is low by construction. A meaningful share are read-only or merely pre-fill forms the user saves; the rest write reversibly (archive, not delete) and audited. Only three tools (†: inventory-reset, process-canon apply, report-liberator’s folder delete) perform higher-impact operations, and each is gated behind explicit user confirmation.
  • The operationally load-bearing ones are specific: the billing flow (cfdi-attacher, invoice-autofill, invoice-auto-regen), order intake (portal-importer, po-reconciler), the part catalog (bulk-upload, process-canon) and shop-floor OEE (paros-linea). Billing in particular is part of why we cannot yet retire the legacy system — so these are where native parity matters most before any transition.
